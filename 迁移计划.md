# CX-Aicode Claude Code -> OpenCode 迁移计划

> 目标：新增 OpenCode 适配层，使核心工作流、只读 agent、`.cx` 状态治理、CodeReview/Verify/Usage 能在 OpenCode 中可用。迁移完成后，业务项目不要求安装 Claude Code；Claude Code 形态仅作为过渡兼容和历史发布资产保留。

## 1. 迁移原则

1. **先兼容，后脱钩**：过渡期保留 `.claude-plugin/`、`commands/`、`agents/`、`hooks/hooks.json`，新增 `.opencode/` 适配层；最终运行路径不得依赖已安装 Claude Code。
2. **核心逻辑不重写**：`.cx` 目录结构、`scripts/agent/*`、`hooks/usage/*`、`scripts/codereview/*`、解析器和报告生成逻辑优先复用。
3. **宿主协议隔离**：把 Claude/OpenCode 的输入输出差异限制在 adapter/wrapper 层，不让业务 hook 和 workflow 逻辑继续散落平台判断。
4. **门禁优先验证**：PreToolUse 等安全门禁是高风险区域，OpenCode 版本必须有独立测试后再默认启用。
5. **文档与命令同步**：所有用户可见命令从 `/cx-aicode:cx-*` 逐步改为 `/cx-*`，同时保留旧格式解析兼容。
6. **OpenCode-only 验收**：必须在没有 `claude` CLI、没有 `~/.claude/plugins/installed_plugins.json`、没有 `CLAUDE_PLUGIN_ROOT` 的环境下完成初始化和主流程。

## 1.1 目标形态

| 阶段 | Claude Code 依赖 | OpenCode 要求 |
| --- | --- | --- |
| 过渡期 | 源码中保留 Claude 文件；不要求业务项目继续安装 Claude Code | `.opencode/` 适配层可运行，核心命令优先走 OpenCode |
| 默认期 | Claude 安装链路变为 legacy fallback | `/cx-init --host=opencode` 成为默认路径 |
| OpenCode-only | 不要求安装 `claude` CLI；不读取 `~/.claude/plugins` 作为主路径 | 所有脚本通过 `CX_AICODE_PLUGIN_ROOT` 或 `.opencode/cx-aicode.json.pluginRoot` 定位插件根 |

## 2. 当前 Claude Code 耦合点

| 模块 | 当前文件 | Claude Code 耦合 |
| --- | --- | --- |
| 插件清单 | `.claude-plugin/plugin.json`, `.claude-plugin/marketplace.json` | 依赖 `claude plugin marketplace/install` |
| 命令 | `commands/*.md` | Claude Code 自动发现 slash command；大量文案使用 `/cx-aicode:cx-*` |
| Agent | `agents/*.md` | frontmatter 使用 `tools`, `model: sonnet`, `maxTurns` 等 Claude 字段 |
| Hook 注册 | `hooks/hooks.json` | 使用 Claude hook event 和 `${CLAUDE_PLUGIN_ROOT}` |

| 插件根定位 | `scripts/lib/resolve-plugin-root.sh`, 多个 `SKILL.md` | 读取 `CLAUDE_PLUGIN_ROOT`, `~/.claude/plugins/installed_plugins.json`, `.claude-plugin/plugin.json` |

## 4. 阶段计划

### P0. 迁移准备

目标：确认 OpenCode 最小运行形态，建立不影响 Claude Code 的适配目录。

任务：

- 新增 `opencode.json`，声明 OpenCode 项目级基础配置。
- 新增 `.opencode/cx-aicode.json`，记录真实 `pluginRoot`、adapter 版本和 host 类型。注意：`OPENCODE_CONFIG_DIR` 只能表示 OpenCode 配置目录，不能直接当成 cx-aicode 源码根。
- 新增 `.opencode/` 目录，但先只放 `cx-help`、`cx-status` 两个低风险命令试点。
- `cx-init` 暂不作为 P0 试点命令；它当前会生成 `.claude/settings.json`、`.claude/hooks/*`、`CLAUDE.md`，必须等 P5 完成 OpenCode 初始化路径后再启用。
- 新增 `scripts/opencode/build-adapter.js`，支持从现有文件复制/转换生成 OpenCode 目录。
- 在 README 增加 OpenCode 试点说明，明确 Claude Code 路径仍受支持。

验收：

- OpenCode 中可看到 `/cx-help` 和 `/cx-status`。
- P0 不会在业务项目中生成 `.claude/*` 或 `CLAUDE.md`。
- 在未安装 Claude Code 的环境中，P0 adapter 仍可生成并被 OpenCode 发现。
- Claude Code 原安装链路暂不主动破坏，但不作为 OpenCode 迁移的必要前置。

### P1. Commands 适配

目标：让 27 个 `cx-*` 命令在 OpenCode 中可见，并正确接收参数。

任务：

- 将 `commands/*.md` 生成到 `.opencode/commands/*.md`。
- 每个命令 prompt 入口补充参数传递说明，确保 `$ARGUMENTS` 被注入。
- 文案中用户可执行命令改为 `/cx-*`。
- Parser 保留旧格式兼容：`/cx-aicode:cx-work` 继续可解析。
- 重点检查以下命令：
  - `cx-init`
  - `cx-work`
  - `cx-codereview`
  - `cx-verify`
  - `cx-finish`
  - `cx-update`
  - `cx-doctor`

验收：

- `/cx-work --spec-id=MES-0535` 在 OpenCode 中能把参数传入流程文本。
- `hooks/usage/command-parser.js` 能同时识别 `/cx-work` 和 `/cx-aicode:cx-work`。
- `cx-help` 输出示例命令全部为 `/cx-*`。

### P2. Skills 适配

目标：让 OpenCode 按自身 skill 目录规则发现核心技能。

任务：

- 将现有 `skills/**/SKILL.md` 映射为 `.opencode/skills/<frontmatter.name>/SKILL.md`。
- 保留正文，清理或忽略 Claude 专用 frontmatter：
  - `allowed-tools`
  - `user-invocable`
- 确保 OpenCode skill 目录名与 frontmatter `name` 一致。
- 对模板和引用文件采用两种策略之一：
  - 短期：在 `.opencode/skills/<name>/` 下复制必要 `templates/`、`references/`。
  - 中期：正文引用统一改为通过 `CX_AICODE_PLUGIN_ROOT` 定位原仓库文件。
- 新增 path/reference 重写清单：扫描 skill 正文中对 `hooks/`、`scripts/`、`skills/`、`templates/`、`reference/` 的相对路径引用，标记为“复制到 skill 包内”或“通过 pluginRoot 解析”。
- 所有依赖 `Skill(...)`、`Agent(...)`、`Task(...)` 的正文示例改为宿主中性描述，并在 OpenCode adapter 中提供等价调用说明。

验收：

- OpenCode 能加载 `cx-work`, `cx-codereview`, `cx-verify`, `tech-vue`, `quality-java` 等核心技能。
- `cx-work` 能读取其 templates。
- `cx-work`, `cx-codereview`, `cx-verify` 中引用的 `scripts/*` 和 `hooks/usage/*` 能通过真实 pluginRoot 找到。
- skill 正文不再要求 Claude 专用 `Skill` 工具作为唯一调用方式。

### P3. Agents 适配

目标：迁移 8 个只读 agent，并保持只读边界。

任务：

- 将 `agents/*.md` 生成到 `.opencode/agents/*.md`。
- 把 Claude frontmatter 转为 OpenCode agent frontmatter。
- 所有只读 reviewer 使用 `mode: subagent`。
- 用 OpenCode `permission` 显式限制：
  - read/list/glob/grep: allow
  - edit/bash/task/skill/external_directory/todowrite/webfetch/websearch: deny
- 不使用 `write: deny` 作为权限字段；OpenCode 的写入类工具由 `edit` permission 覆盖。
- 模型字段从 `sonnet/opus/haiku` 抽象名改为 OpenCode 可识别的 provider/model 配置，或先移除让用户配置默认模型。
- 更新技能正文中关于 `Agent({ subagent_type: ... })` 的示例，加入 OpenCode 调用说明或中性描述。

验收：

- 8 个 agent 在 OpenCode 中可见。
- reviewer agent 无法修改文件、执行 bash、再派生子 agent。
- 现有 `tests/agents/*.js` 新增 OpenCode frontmatter 校验。

### P4. Hooks / Plugin 适配

目标：把 Claude hook manifest 改造为 OpenCode plugin wrapper，并复用现有门禁逻辑。

任务：

- 新增 `.opencode/plugins/cx-aicode.js`。
- 在 wrapper 中监听 OpenCode 事件，并映射到现有 dispatcher 能理解的统一事件对象。
- 新增平台无关 hook adapter：

```text
hooks/adapters/
├── claude.js       # Claude payload -> normalized event / normalized output -> Claude output
├── opencode.js     # OpenCode event -> normalized event / normalized output -> OpenCode behavior
└── normalized.js   # 统一事件 schema
```

建议统一 schema：

```json
{
  "event": "pre-tool-use",
  "toolName": "Edit",
  "toolInput": {
    "file_path": "src/app.js"
  },
  "prompt": "",
  "sessionId": "",
  "projectRoot": ""
}
```

事件映射：

| Claude Code | OpenCode | 迁移动作 |
| --- | --- | --- |
| `SessionStart` | `session.created` | 输出 silent/context 初始化 |
| `UserPromptSubmit` | 无稳定 1:1 等价；先用 `command.executed` 覆盖 slash command，普通 prompt 只做 best-effort 观察事件验证 | usage collect / spec-id gate 降级 |
| `PreToolUse` | `tool.execute.before` | permission gate、编码 gate、三文件 gate |
| `PostToolUse` | `tool.execute.after`, `file.edited` | usage transition、work event queue |
| `PostToolUseFailure` | `tool.execute.after` error 或 `session.error` | error recovery |
| `PermissionDenied` | `permission.replied` | error recovery / diagnostics |
| `Stop` | `session.idle` 或 session stop 类事件 | session close context |

注意：

- Claude 的 `permissionDecision: deny` 在 OpenCode 中不能原样输出；OpenCode wrapper 应改为抛错或返回宿主要求的 permission 结果。
- Claude 的 `systemMessage` 注入机制需确认 OpenCode 是否等价；没有等价能力时，降级为日志或 command 输出提醒。
- `UserPromptSubmit` 在 OpenCode 没有稳定 1:1 等价能力。不能把 spec-id 强门禁放在普通 prompt 事件上；强门禁必须落到 command template preflight、skill 入口 preflight、`tool.execute.before` 三处。
- usage 用户请求统计先确保 slash command 触发统计；普通 prompt 恢复提醒和统计作为次优先级，待实际事件验证后再启用。

验收：

- OpenCode 执行写文件前会触发设计审批 gate。
- GBK/非 UTF-8 编辑 gate 仍能 deny。
- `.cx/{SPEC-ID}/work/task_plan.md` 编辑后 PostTool 仍能入队 work event。
- OpenCode hook 测试覆盖 allow/deny/require-approval/error 路径。

### P5. 初始化、更新、诊断适配

目标：让 `/cx-init`、`/cx-update`、`/cx-doctor` 不再只认识 Claude Code。

任务：

- `resolve-plugin-root.sh` 新增解析顺序：
  1. `CX_AICODE_PLUGIN_ROOT`
  2. `.opencode/cx-aicode.json` 或 `opencode.json` 中的 `pluginRoot`
  3. 当前源码根
  4. OpenCode adapter 目录对应的真实 pluginRoot 配置
  5. Claude 旧路径 fallback（legacy only，不作为 OpenCode 主路径）
- `OPENCODE_CONFIG_DIR` 只用于定位 OpenCode 配置目录和 adapter 文件，不作为真实 pluginRoot；除非其中显式存在 `.opencode/cx-aicode.json` 并声明 `pluginRoot`。
- `init-project.ps1` 新增 `-Host OpenCode|Claude|Auto` 参数；当未检测到 Claude Code 且存在 OpenCode 时，`Auto` 必须选择 OpenCode。
- OpenCode 模式下生成：
  - `.cx/`
  - `opencode.json` 或合并已有配置
  - `.opencode/cx-aicode.json`，写入真实 pluginRoot 和 adapter 版本
  - `.opencode/plugins/cx-aicode.js`
  - `.opencode/commands/*` 或指向统一配置目录
  - `.opencode/agents/*`
  - `.opencode/skills/*`
  - `AGENTS.md`
- Claude 模式继续生成：
  - `.claude/settings.json`
  - `.claude/hooks/*`
  - `CLAUDE.md`
- `cx-update` 改为双宿主：
  - Claude：legacy 模式继续读 `~/.claude/plugins/installed_plugins.json`
  - OpenCode：优先读 `OPENCODE_CONFIG_DIR` 下的 `.opencode/cx-aicode.json.pluginRoot`，或管理员显式指定的 pluginRoot
- 所有脚本在未安装 `claude` CLI 时不得失败，除非用户显式选择 `--host=claude`。
- `cx-doctor` 增加 `--host=opencode` 检查：
  - OpenCode CLI 是否存在
  - `.opencode/commands`
  - `.opencode/agents`
  - `.opencode/skills`
  - `.opencode/plugins/cx-aicode.js`
  - `opencode.json`

验收：

- `/cx-init --host=opencode --tech=vue` 在 OpenCode 项目中生成可运行配置。
- `/cx-doctor --host=opencode` 能报告 commands/agents/skills/plugins 状态。
- `/cx-init --host=opencode --dry-run` 不会生成 `.claude/*` 或 `CLAUDE.md`。
- `/cx-update --check` 不依赖 `~/.claude/plugins/installed_plugins.json`。
- 删除或隐藏 `claude` CLI、`~/.claude/plugins/installed_plugins.json` 后，OpenCode 初始化、诊断、核心命令仍可运行。

## 5. 建议优先级

| 顺序 | 范围 | 原因 |
| --- | --- | --- |
| 1 | `.opencode/commands`, `cx-help`, `cx-status` | 最小可见性和只读验证；避免 P0 误生成 Claude 配置 |
| 2 | commands 参数传递 | 没有 `$ARGUMENTS`，`--spec-id` 会丢失 |
| 3 | skills 路径和引用重写 | skill 可见不等于可运行；必须先解决模板和脚本定位 |
| 4 | agents 只读权限 | 防止 reviewer 在 OpenCode 中获得写权限 |
| 5 | pre-tool gate | 安全门禁是核心价值和高风险点 |
| 6 | init/update/doctor | 解决真实项目落地；此时再启用 `cx-init` |
| 7 | OpenCode-only 脱钩验证 | 确认无 `claude` CLI、无 `~/.claude/plugins` 时仍可运行 |
| 8 | usage 统计 | 依赖事件准确性，可在主流程稳定后补强 |
| 9 | 全量文档和回归 | 发布前收尾 |

## 6. 风险与处理

| 风险 | 影响 | 处理 |
| --- | --- | --- |
| OpenCode 没有等价 `UserPromptSubmit` | 恢复提醒和 usage 用户请求统计不完整；spec-id prompt gate 不能照搬 | 强门禁落到 command template、skill preflight、`tool.execute.before`；先用 `command.executed` 覆盖 slash command，普通 prompt 功能降级 |
| OpenCode hook deny 机制不同 | 门禁失效或报错体验差 | adapter 层单独实现 deny/throw/permission 返回，并做 e2e 验证 |
| Agent 工具权限字段不兼容 | 只读 reviewer 可能获得写权限 | 使用 OpenCode 官方 permission key；写入类统一通过 `edit: deny` 控制 |
| `OPENCODE_CONFIG_DIR` 被误当 pluginRoot | `cx-init/cx-work/cx-codereview` 找不到模板和脚本 | `OPENCODE_CONFIG_DIR` 仅定位 adapter；真实源码根必须来自 `CX_AICODE_PLUGIN_ROOT` 或 `.opencode/cx-aicode.json.pluginRoot` |
| 插件根定位失败 | `cx-init/cx-work/cx-codereview` 找不到模板和脚本 | 全部脚本优先支持 `CX_AICODE_PLUGIN_ROOT` 和显式 `pluginRoot` |
| 仍隐式依赖 Claude 安装目录 | OpenCode-only 环境无法运行 | 增加 no-Claude 验收：隐藏 `claude` CLI、清空 `CLAUDE_PLUGIN_ROOT`、移除 `~/.claude/plugins` 后跑初始化和核心命令 |
| 双宿主文档混乱 | 用户不知道使用哪套命令 | 文档按 Claude Code/OpenCode 分章，默认示例统一 `/cx-*` |
| 更新模型不同 | `/cx-update` 误写 Claude cache | OpenCode 模式必须显式 pluginRoot，不自动写 `~/.claude`；Claude cache 只在 legacy 模式使用 |
