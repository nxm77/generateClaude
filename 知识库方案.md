# 企业知识库建设方案

> 适用场景：企业内部知识库建设，兼顾**人的阅读、检索、培训、审计**需求，同时为 AI 编程、CodeReview、规范问答、项目知识检索提供结构化知识来源。  
> 重点约束：优先选择开源授权清晰、适合私有化部署、便于长期维护、能与 Git / AI 编程工具链集成的软件。

---

## 1. 方案结论

综合开源授权、人的阅读体验、全文检索、版本管理、AI 编程适配能力，推荐采用如下组合：

```text
主知识库源头：Git + Markdown / MDX
人类阅读门户：Docusaurus，备选 MkDocs
轻量全文检索：Pagefind
企业级全文检索：OpenSearch
AI 编程代码知识库：Repomix + /cx-codebase-skill
AI/RAG 扩展层：Haystack 或 RAGFlow
向量检索层：Qdrant 或 pgvector
辅助知识平台：BookStack，可选，用于非研发人员在线编辑类知识
```

如果只能确定一个主平台，建议选择：

> **Docusaurus 作为企业知识库主门户，Git/Markdown 作为知识源，Pagefind/OpenSearch 作为检索层，Repomix + Haystack/RAGFlow 作为 AI 编程扩展。**

---

## 2. 为什么主知识库不建议直接选择 AI 知识库平台

企业知识库首先要解决的是：

1. 人能不能系统性阅读；
2. 能不能按目录、主题、项目、技术栈查找；
3. 能不能进行版本管理和变更审核；
4. 能不能长期维护，不变成“上传文件垃圾堆”；
5. 能不能作为 AI 检索和上下文生成的可信知识源。

RAGFlow、Dify、FastGPT、MaxKB 这类平台更偏向“问答型知识库”或“AI 应用平台”。它们适合做 AI 问答入口，但不适合作为企业主知识库的唯一源头。

正确思路应是：

```text
先建设人可读、可维护、可审计的主知识库
        ↓
再从主知识库同步数据给全文检索和 AI/RAG 系统
        ↓
最后让 AI 编程工具调用经过治理的知识
```

## 3. 推荐总体架构

本方案必须把“知识展示”和“访问授权”分开设计：Docusaurus/MkDocs 负责生成可阅读门户，**统一认证网关、Git 权限、搜索权限和 RAG 权限**共同负责授权控制。

## 4. 核心选型建议

### 4.1 主知识库门户：Docusaurus

**推荐等级：首选**

Docusaurus 适合承担企业知识库主门户，原因如下：

1. **授权友好**  
   Docusaurus 使用 MIT 许可证，适合企业内部私有化使用、二次集成和长期维护。

2. **适合人阅读**  
   它本质是文档站生成器，天然支持目录、侧边栏、文档分组、版本化、主题定制、搜索集成，适合建设企业级知识门户。

3. **适合文档即代码**  
   所有知识以 Markdown/MDX 形式进入 Git，便于评审、追踪、回滚、分支管理和自动发布。

4. **适合 AI 编程知识库**  
   编码规范、评分标准、cx-aicode 使用手册、SOP、Agents、Skills、Hooks、项目 reference 都可以以 Markdown 形式维护，后续可直接作为 AI/RAG 的数据源。

5. **适合多版本规范管理**  
   企业 AI 编程规范、插件版本、项目规范会持续演进，Docusaurus 的版本化能力适合保留历史规范。

适合承载内容：

```text
- 企业 AI 编程 SOP
- cx-aicode 插件使用手册
- /cx-spec、/cx-brainstorm、/cx-work、/cx-codereview、/cx-verify、/cx-test 命令说明
- Agents / Skills / Hooks 管理规范
- 编码规范
- CodeReview 评分标准
- Verify / Test 规则
- 技术栈知识库：C++、Java、Vue、VB.NET、C#、Python、Go
- 项目 reference
- 项目业务知识
- 常见问题 FAQ
- 培训材料
```

---

### 4.2 备选主门户：MkDocs

**推荐等级：备选，适合轻量场景**

MkDocs 的优势是简单、轻量、Markdown 原生、Python 生态友好，BSD 授权也较适合企业使用。

适合以下情况：

1. 不需要复杂前端定制；
2. 文档以 Markdown 为主；
3. 希望部署简单；
4. 团队更熟悉 Python；
5. 主要用于项目规范、技术栈手册、运维文档。

如果未来知识库会发展成企业级门户，包含多个技术栈、多个项目、培训中心、规范版本、插件手册、AI 编程专区，Docusaurus 更合适。

---

### 4.3 轻量全文检索：Pagefind

**推荐等级：第一阶段首选**

Pagefind 适合静态站点搜索，不需要单独部署后端搜索服务。

适合场景：

1. 文档数量在几百到几千篇；
2. 不需要复杂权限过滤；
3. 希望搜索能力快速上线；
4. 希望降低运维复杂度；
5. Docusaurus/MkDocs 构建完成后直接生成搜索索引。

限制：

1. 不适合复杂权限控制；
2. 不适合超大规模跨系统搜索；
3. 不适合对搜索结果做复杂排序、字段过滤、权限过滤。

建议：第一阶段使用 Pagefind，等规模变大后再引入 OpenSearch。

---

### 4.4 企业级全文检索：OpenSearch

**推荐等级：第二阶段引入**

OpenSearch 适合企业级全文检索和搜索分析场景。

适合场景：

1. 文档数量大；
2. 多项目、多部门、多知识源；
3. 需要复杂搜索条件；
4. 需要按项目、技术栈、文档类型、版本、有效期过滤；
5. 需要与权限系统、SSO、审计系统集成；
6. 需要统一检索文档、代码摘要、报告、FAQ、日志、变更记录。

建议：

- 第一阶段不要直接上 OpenSearch，避免系统过重；
- 当 Pagefind 无法满足权限、规模和跨系统检索要求时，再升级到 OpenSearch；
- OpenSearch 索引中必须写入权限元数据、项目元数据、版本元数据和文档状态字段。

---

### 4.5 AI 编程代码知识库：Repomix

**推荐等级：AI 编程代码上下文首选**

Repomix 适合将代码仓库打包成 AI-friendly 文件，是 AI 编程知识库中处理代码上下文的核心工具。

适合用途：

1. 生成项目级代码上下文；
2. 生成模块摘要；
3. 生成关键文件索引；
4. 支持 `/cx-codebase-skill` 生成项目级 Codebase Skill；
5. 为 Claude Code CLI / cx-aicode 提供稳定上下文；
6. 避免每次让 AI 盲目扫描全仓库，降低 Token 消耗。

建议定位：

```text
Docusaurus / MkDocs 负责“文档知识”
Repomix 负责“代码上下文知识”
Haystack / RAGFlow 负责“AI 检索与问答”
```

---

### 4.6 AI/RAG 扩展层：Haystack 或 RAGFlow

#### 方案 A：Haystack

**推荐等级：长期自研底座首选**

Haystack 适合企业自研可控的 RAG 管线。

优点：

1. Apache 2.0 授权，适合企业深度集成；
2. 适合构建自定义检索、重排、路由、问答、Agent 流程；
3. 便于与 ai-switch、cx-aicode、权限系统、审计系统集成；
4. 对企业长期掌控技术栈更有利。

缺点：

1. 不是开箱即用平台；
2. 需要研发团队建设检索管线和应用层；
3. 前期落地速度慢于 RAGFlow。

适合你们长期方向：

```text
企业自研 AI 编程知识服务
企业统一知识检索 API
与 cx-aicode / ai-switch / OTel 统计系统深度集成
```

#### 方案 B：RAGFlow

**推荐等级：快速落地首选**

RAGFlow 适合快速搭建文档问答、知识库检索、引用溯源和 RAG 应用。

优点：

1. Apache 2.0 授权；
2. 平台能力较完整；
3. 更适合快速试点；
4. 文档解析、知识库、问答链路比纯框架更开箱即用。

缺点：

1. 运维组件更多；
2. 深度定制不如 Haystack 灵活；
3. 与企业现有 AI 编程流程集成时仍需二次开发。

建议：

- 若希望快速试点：先上 RAGFlow；
- 若希望长期做企业知识服务底座：建设 Haystack 管线；
- 两者不是互斥关系，可以第一阶段 RAGFlow，第二阶段抽象内部知识检索 API，逐步替换或并存。

---

### 4.7 向量库：Qdrant / pgvector

#### Qdrant

适合中大型企业 RAG 场景。

优点：

1. Apache 2.0 授权；
2. 部署相对简单；
3. 过滤能力较好；
4. 适合将文档向量与元数据一起检索。

#### pgvector

适合已有 PostgreSQL 的企业。

优点：

1. 运维成本低；
2. 元数据和向量可放在同一个数据库；
3. 适合第一阶段或中小规模知识库。

建议：

```text
第一阶段：pgvector 或 Qdrant
第二阶段：若知识量和并发明显扩大，再评估 Milvus 或更复杂架构
```

---

### 4.8 辅助平台：BookStack

**推荐等级：可选，不作为 AI 编程主知识库**

BookStack 授权友好，阅读体验和在线编辑体验较好，适合业务部门、运维团队、非研发人员维护普通知识。

适合内容：

1. 业务操作手册；
2. 运维值班手册；
3. 部门制度；
4. 普通 FAQ；
5. 非研发人员维护的知识。

不建议作为 AI 编程主知识库的原因：

1. Git 版本管理能力弱于 Docusaurus/MkDocs；
2. MR/PR 审核不如 Git 文档流程自然；
3. 与代码仓库、插件、reference、skills 的联动不如文档即代码方式；
4. AI 可消费性需要额外导出或同步。

推荐定位：

```text
Docusaurus：研发规范、AI 编程、项目 reference、技术栈知识
BookStack：业务部门在线维护类知识
```

## 5. 不建议作为主知识库的选型

### 5.1 Dify

Dify 是功能完整的 AI 应用开发平台，但其许可证是基于 Apache 2.0 的修改版并带附加条件。对于企业内部快速试点 AI 应用可以考虑，但不建议作为企业知识库主底座。

主要原因：

1. 授权不是标准 Apache 2.0；
2. 更偏 AI 应用平台，不是人类主阅读门户；
3. 企业若深度定制、二次封装或平台化推广，需要法务确认边界。

### 5.2 MaxKB

MaxKB 中文体验较好，适合快速搭建知识问答系统，但 GPL-3.0 对企业二次开发、分发和集成有更强约束。

不建议作为主底座的原因：

1. GPL-3.0 授权风险需要法务评估；
2. 偏 AI 问答，不是文档治理主系统；
3. 作为企业 AI 编程基础设施底座不够稳妥。

### 5.3 Wiki.js

Wiki.js 功能强，但 AGPL-v3 对网络部署和二次开发更敏感。

不建议作为主底座的原因：

1. AGPL-v3 授权约束强；
2. 企业内部深度集成时合规复杂度较高；
3. 不如 Git + Docusaurus/MkDocs 适合研发知识库和 AI 编程知识库。

### 5.4 直接用 RAG 平台替代知识库

不建议直接将 RAGFlow、Dify、FastGPT、MaxKB 当成唯一知识库，因为这容易导致：

1. 文档没有清晰目录；
2. 知识缺少版本管理；
3. 知识变更没有审核流程；
4. 人员培训依赖问答，不利于系统学习；
5. 上传文件越来越多，但没人知道哪些过期；
6. AI 回答依赖检索结果，缺少统一可信源。

## 7. 文档元数据规范

每篇文档建议增加 Front Matter，便于搜索、权限、版本、AI 检索和过期治理。

```yaml
---
title: Java 编码规范
owner: 架构组
domain: ai-programming
category: coding-rule
project: common
tech_stack: java
version: v1.0
status: active
valid_from: 2026-07-01
valid_until:
review_cycle: quarterly
confidential_level: L1
allowed_groups:
  - dev
  - qa
  - architecture
denied_groups: []
access_policy: dev-common-default
search_index: true
ai_consumable: true
rag_index: true
last_reviewed: 2026-07-01
---
```

字段说明：

| 字段 | 含义 |
|---|---|
| title | 文档标题 |
| owner | 责任部门或责任人 |
| domain | 所属领域 |
| category | 文档类型 |
| project | 所属项目，common 表示通用 |
| tech_stack | 技术栈 |
| version | 文档版本 |
| status | active / draft / deprecated / archived |
| valid_from | 生效日期 |
| valid_until | 失效日期，可为空 |
| review_cycle | 复审周期 |
| confidential_level | 权限级别，建议使用 L0~L5 |
| allowed_groups | 允许访问的用户组 |
| denied_groups | 明确禁止访问的用户组 |
| access_policy | 权限策略名称 |
| search_index | 是否进入全文检索索引 |
| ai_consumable | 是否允许 AI 使用 |
| rag_index | 是否进入 RAG 索引 |
| last_reviewed | 最近复审日期 |

## 8. 检索设计

### 8.1 人类检索

第一阶段使用：

```text
Docusaurus / MkDocs + Pagefind
```

搜索能力：

1. 标题搜索；
2. 全文搜索；
3. 按目录导航；
4. 按技术栈导航；
5. 按项目导航；
6. 按文档类型导航；
7. 按 FAQ / SOP / 规范分类查看。

第二阶段升级：

```text
Docusaurus / MkDocs + OpenSearch
```

增强能力：

1. 权限过滤；
2. 项目过滤；
3. 技术栈过滤；
4. 文档状态过滤；
5. 版本过滤；
6. 有效期过滤；
7. 搜索日志分析；
8. 热门问题统计；
9. 搜索无结果词分析。

### 8.2 AI 检索

AI 检索不应直接抓取所有页面，而应经过导出和过滤：

```text
Git 文档源
  ↓
校验 Front Matter
  ↓
过滤 ai_consumable=true 且 rag_index=true 的文档
  ↓
转换为结构化 Markdown / JSON
  ↓
切分、向量化、写入 Qdrant/pgvector
  ↓
Haystack/RAGFlow 提供检索 API
  ↓
cx-aicode / ai-switch / Claude Code CLI 调用
```

## 10. 权限与访问控制设计

本方案需要明确：**Docusaurus/MkDocs 只是静态文档门户，不是权限系统**。企业知识库访问需要授权时，不能只依赖前端隐藏菜单、目录隐藏或搜索隐藏，而要在源头、发布、网关、搜索、RAG、向量库六个层面共同控制。

推荐总体原则：

```text
源头有权限，发布分权限，入口要认证，搜索不越权，AI 不绕权，访问可审计
```

### 10.1 权限控制总体架构

```text
用户
  ↓
统一身份认证
SSO / LDAP / AD / OIDC / OAuth2 Proxy / Nginx
  ↓
权限判定
用户 → 部门 / 项目组 / 角色 / 安全级别
  ↓
知识门户访问
Docusaurus / MkDocs 分区站点或分区路径
  ↓
搜索访问
Pagefind 分权限域索引，或 OpenSearch 文档级权限过滤
  ↓
AI/RAG 访问
Haystack / RAGFlow 检索时按 user / group / project / role 过滤
  ↓
审计日志
记录访问、搜索、问答、索引命中文档、引用来源
```

### 10.2 推荐权限分级

建议将知识库分为 6 个访问级别。

| 级别 | 名称 | 访问范围 | 是否允许进入 RAG | 示例 |
|---|---|---|---|---|
| L0 | 内部公开知识 | 全体员工 | 可选 | 通用制度、公开培训材料 |
| L1 | 研发通用知识 | 所有研发人员 | 是 | AI 编程 SOP、cx-aicode 使用手册 |
| L2 | 技术栈知识 | 对应技术栈团队 | 是 | Java/Vue/C++/Python 规范 |
| L3 | 项目知识 | 项目组授权人员 | 是，但必须项目过滤 | MES、EAP、QMS 项目文档 |
| L4 | 敏感知识 | 特定角色/负责人 | 谨慎，默认不进 RAG | 架构、安全、生产运维、接口细节 |
| L5 | 禁止 AI 使用知识 | 少数授权人员 | 否 | 密码、密钥、客户隐私、生产账号、敏感清单 |

建议采用默认保守策略：

```text
没有明确 allowed_groups 的文档不得发布到受限区
没有明确 ai_consumable=true 的文档不得进入 RAG
confidential_level=L4/L5 的文档默认不得进入 RAG
```

### 10.3 Git 源头权限

知识源头必须受控。推荐在 GitLab/GitHub Enterprise/内部 Git 中按目录、项目或仓库授权。

示例：

```text
knowledge-base/
  ├─ docs/common/              # 所有人可读，平台组可写
  ├─ docs/ai-programming/      # 研发可读，平台组/架构组可写
  ├─ docs/tech-stack/java/     # Java 技术栈成员可维护
  ├─ docs/tech-stack/cpp/      # C++ 技术栈成员可维护
  ├─ docs/projects/mes/        # MES 项目组可读写
  ├─ docs/projects/eap/        # EAP 项目组可读写
  ├─ docs/security/            # 架构组/安全组可读写
  └─ docs/restricted/          # 特殊授权，不进入统一公开门户
```

源头权限建议：

1. 通用文档可以放在统一知识库仓库；
2. 项目敏感文档可按项目拆分仓库；
3. CI 构建时根据文档元数据和仓库权限生成不同站点或不同索引；
4. 项目级 Codebase Skill、Repomix 输出文件必须继承项目仓库权限；
5. 禁止把受限项目文档复制到 common 目录或公共搜索索引。

### 10.4 阅读门户权限

Docusaurus/MkDocs 静态站推荐放在统一认证网关后面：

```text
用户浏览器
  ↓
企业 SSO / LDAP / AD / OIDC
  ↓
OAuth2 Proxy / Nginx / Ingress
  ↓
Docusaurus / MkDocs 静态站
```

访问控制可以采用两种模式。

#### 模式 A：按权限域拆站点，推荐第一阶段采用

```text
kb-common.company.local        # 全体员工
kb-ai.company.local            # 研发人员
kb-java.company.local          # Java 技术栈团队
kb-cpp.company.local           # C++ 技术栈团队
kb-mes.company.local           # MES 项目组
kb-eap.company.local           # EAP 项目组
kb-security.company.local      # 架构/安全组
```

优点：

1. 权限边界清晰；
2. 静态搜索索引不会混入无权限内容；
3. 运维和审计简单；
4. 不需要在 Docusaurus/MkDocs 内做复杂二次开发。

缺点：

1. 多个站点需要统一导航；
2. 跨权限域搜索需要后续引入 OpenSearch。

#### 模式 B：单门户 + 后端权限过滤，适合第二阶段

```text
kb.company.local
  ├─ /common
  ├─ /ai-programming
  ├─ /tech-stack/java
  ├─ /projects/mes
  └─ /security
```

该模式需要网关、后端或 OpenSearch 做细粒度权限控制。若只是静态站，不建议一开始采用这种模式承载敏感内容。

### 10.5 Pagefind 搜索权限

Pagefind 适合同一权限域内的静态全文检索。关键限制是：**不能把不同权限级别的文档构建到同一个 Pagefind 索引中**。

错误做法：

```text
公共文档 + MES 项目文档 + 安全文档 → 一个 Pagefind 索引
```

风险：

1. 用户可能通过搜索摘要看到无权限内容；
2. 页面入口隐藏不代表搜索索引安全；
3. 静态索引一旦发布，无法按用户动态过滤。

正确做法：

```text
kb-common → pagefind-common 索引
kb-ai     → pagefind-ai 索引
kb-mes    → pagefind-mes 索引
kb-eap    → pagefind-eap 索引
```

结论：

| 场景 | 是否适合 Pagefind |
|---|---|
| 单权限域知识库 | 适合 |
| 项目内部知识库 | 适合 |
| 多项目统一门户但无敏感隔离 | 可用 |
| 多项目、多角色、文档级权限 | 不适合，应使用 OpenSearch |

### 10.6 OpenSearch 搜索权限

当需要统一搜索多个项目、多个技术栈和多个权限域时，应引入 OpenSearch。

OpenSearch 索引中每篇文档必须写入权限元数据：

```json
{
  "doc_id": "mes-architecture-overview-v1",
  "title": "MES 架构说明",
  "project": "mes",
  "tech_stack": "cpp",
  "confidential_level": "L3",
  "allowed_groups": ["mes-dev", "architecture"],
  "denied_groups": [],
  "status": "active",
  "version": "v1.0",
  "rag_index": true,
  "ai_consumable": true
}
```

搜索时必须把用户权限转换为过滤条件：

```text
user = 张三
user_groups = [dev, mes-dev]
search_filter = allowed_groups contains dev or mes-dev
                AND status = active
                AND confidential_level <= 用户最高可访问级别
```

要求：

1. 搜索 API 不接受前端直接传入 allowed_groups；
2. allowed_groups 必须由服务端根据登录用户身份生成；
3. 搜索结果、摘要、高亮片段都必须经过权限过滤；
4. 搜索日志记录用户、关键词、命中文档、访问结果；
5. deprecated/archived 文档默认不进入普通搜索结果，可在历史模式中查看。

### 10.7 AI/RAG 权限

AI/RAG 权限是整个方案中最容易出问题的地方。必须避免：

```text
用户没有项目文档权限，但通过 AI 问答获得了项目设计、接口、生产规则
```

RAG 检索必须遵守以下流程：

```text
用户登录
  ↓
服务端获取 user_id / groups / projects / roles / max_confidential_level
  ↓
构造检索过滤条件
  ↓
只召回用户有权访问的 chunk
  ↓
只基于有权限 chunk 生成回答
  ↓
回答中展示引用来源
  ↓
记录审计日志
```

RAG chunk 元数据建议包含：

```json
{
  "chunk_id": "mes-architecture-overview-v1-0003",
  "doc_id": "mes-architecture-overview-v1",
  "source_path": "docs/projects/mes/architecture/overview.md",
  "project": "mes",
  "tech_stack": "cpp",
  "confidential_level": "L3",
  "allowed_groups": ["mes-dev", "architecture"],
  "status": "active",
  "version": "v1.0",
  "ai_consumable": true,
  "rag_index": true
}
```

RAG 权限要求：

1. `ai_consumable=false` 的文档不得进入向量库；
2. `rag_index=false` 的文档不得进入向量库；
3. L4 文档默认不进入 RAG，确需进入时必须单独审批；
4. L5 文档禁止进入 RAG；
5. RAG 检索必须使用服务端权限过滤，不能只依赖前端；
6. prompt 中不得拼接用户无权限内容；
7. AI 回答必须带引用，便于核验是否越权；
8. 对敏感知识问答要记录审计日志。

### 10.8 向量库与底层服务权限

Qdrant、pgvector、OpenSearch 等底层服务不能直接暴露给普通用户或前端。

推荐部署原则：

```text
前端 / CLI / 插件
  ↓
知识检索 API
  ↓
权限过滤
  ↓
OpenSearch / Qdrant / pgvector
```

要求：

1. 向量库只允许内网访问；
2. 使用 API Key、数据库账号或服务账号控制访问；
3. 禁止前端直接访问向量库；
4. 禁止把向量库管理密钥写入客户端配置；
5. 不同环境 dev/test/prod 使用不同密钥；
6. 向量库备份也要按同等级别保护；
7. RAG 导出文件和中间 JSON 文件也必须按权限保护。

### 10.9 文档元数据中的权限字段

建议在 Front Matter 中补充权限字段：

```yaml
---
title: MES 架构说明
owner: MES 项目组
domain: project
category: architecture
project: mes
tech_stack: cpp
version: v1.0
status: active
confidential_level: L3
allowed_groups:
  - mes-dev
  - architecture
denied_groups: []
ai_consumable: true
rag_index: true
search_index: true
access_policy: project-mes-default
last_reviewed: 2026-07-01
---
```

字段说明：

| 字段 | 含义 |
|---|---|
| confidential_level | 知识敏感级别，L0~L5 |
| allowed_groups | 允许访问的用户组 |
| denied_groups | 明确禁止访问的用户组，优先级高于 allowed_groups |
| ai_consumable | 是否允许 AI 使用 |
| rag_index | 是否进入 RAG 索引 |
| search_index | 是否进入全文检索索引 |
| access_policy | 权限策略名称，便于统一治理 |

### 10.10 审计与日志

知识库访问需要记录审计日志，尤其是受限文档、搜索和 AI 问答。

建议记录：

| 类型 | 记录内容 |
|---|---|
| 门户访问 | 用户、时间、页面、IP、权限域 |
| 搜索行为 | 用户、关键词、命中文档、点击文档 |
| RAG 问答 | 用户、问题、召回文档、引用来源、模型、回答时间 |
| 索引变更 | 文档 ID、版本、状态、权限元数据、索引时间 |
| 权限拒绝 | 用户、请求资源、拒绝原因 |

审计日志用途：

1. 排查越权访问；
2. 分析热门知识和无结果搜索；
3. 追踪 AI 回答引用来源；
4. 发现过期知识仍被访问；
5. 支撑安全合规检查。

### 10.11 第一阶段推荐权限落地方案

第一阶段不建议直接实现复杂文档级权限，建议采用“分区站点 + 网关认证 + 分区索引”的方式。

```text
GitLab 权限
  ↓
CI 按权限域构建多个 Docusaurus/MkDocs 站点
  ↓
Nginx / OAuth2 Proxy 接入企业 SSO
  ↓
每个站点生成独立 Pagefind 索引
  ↓
RAG 只接入 L0~L3 且 ai_consumable=true 的文档
```

第一阶段站点建议：

```text
kb-common     # 全体员工可访问
kb-ai-dev     # 研发人员可访问
kb-techstack  # 技术栈团队可访问，可再细分
kb-project-*  # 每个项目一个站点或一个权限域
kb-security   # 特殊权限，默认不进 RAG
```

优点：

1. 权限边界简单可靠；
2. 适合静态站；
3. Pagefind 不会泄露跨权限域内容；
4. 容易与现有 GitLab 项目组权限对齐；
5. 后续可以平滑升级到 OpenSearch 统一搜索。

### 10.12 第二阶段推荐权限落地方案

第二阶段在权限模型稳定后，再做统一搜索和统一 AI 知识服务。

```text
统一 SSO
  ↓
知识门户
  ↓
知识检索 API
  ├─ OpenSearch：全文检索 + 文档级权限过滤
  ├─ Qdrant/pgvector：向量检索 + 元数据过滤
  └─ 审计服务：记录搜索与问答行为
  ↓
ai-switch / cx-aicode / Claude Code CLI
```

第二阶段目标：

1. 一个统一搜索入口；
2. 用户只看到自己有权限的知识；
3. AI 只召回用户有权限的知识；
4. 支持项目、技术栈、版本、状态过滤；
5. 搜索和 AI 问答都有审计记录。

### 10.13 权限风险与防护

| 风险 | 表现 | 防护 |
|---|---|---|
| 静态站泄露 | 无权限页面被直接访问 | 网关认证、分站点部署、禁止公开发布 |
| 搜索索引泄露 | Pagefind 索引包含受限内容 | 每个权限域单独构建索引 |
| RAG 越权 | AI 回答了无权限项目知识 | 检索 API 服务端权限过滤 |
| 向量库泄露 | 用户直接查询 Qdrant/pgvector | 向量库内网隔离，禁止前端直连 |
| 元数据错误 | allowed_groups 填错导致越权 | CI 校验、owner 审核、权限策略模板 |
| 复制粘贴泄露 | 受限内容被放到 common 区 | MR 审核、敏感词扫描、目录权限检查 |
| 旧索引残留 | 权限变更后旧索引仍可查 | 索引重建、索引版本切换、缓存清理 |

## 11. 知识治理流程

### 11.1 文档新增流程

```text
提交 Markdown 文档
  ↓
填写 Front Matter
  ↓
本地预览
  ↓
提交 Merge Request
  ↓
责任人审核
  ↓
CI 校验格式、链接、元数据
  ↓
合并到主分支
  ↓
自动发布文档站
  ↓
自动更新搜索索引
  ↓
按规则同步到 RAG 索引
```

### 11.2 文档修改流程

```text
修改文档
  ↓
说明变更原因
  ↓
更新 version / last_reviewed
  ↓
MR 审核
  ↓
发布新版本
  ↓
旧版本保留或标记 deprecated
```

### 11.3 文档过期治理

每月或每季度运行检查脚本：

1. 查找超过 review_cycle 未复审的文档；
2. 查找 valid_until 已过期的文档；
3. 查找 owner 为空的文档；
4. 查找 status=draft 长期未发布的文档；
5. 查找 rag_index=true 但 ai_consumable=false 的冲突文档；
6. 生成知识库健康报告。

## 13. 实施路线图

### 第一阶段：基础知识门户建设，2~4 周

目标：先让人能阅读、检索、维护。

任务：

1. 建立 Git 知识库仓库；
2. 搭建 Docusaurus 或 MkDocs；
3. 建立目录结构；
4. 导入现有 SOP、reference、编码规范、评分标准；
5. 接入 Pagefind；
6. 配置内网访问；

## 14. 风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| 知识无人维护 | 文档过期、重复、冲突 | 建立 owner、review_cycle、过期扫描 |
| 搜索结果不准 | 找不到、找错版本 | 元数据规范、标题规范、OpenSearch 升级 |
| AI 引用过期知识 | RAG 使用 deprecated 文档 | rag_index 和 status 双重过滤 |
| 权限泄露 | AI 检索到不该看的内容 | 建立 allowed_groups 和 AI 检索权限过滤 |
| 授权风险 | 引入 GPL/AGPL 或附加限制软件 | 首选 MIT/BSD/Apache 2.0，法务确认例外组件 |
| 文档站不适合非研发人员编辑 | 非技术用户不愿提交 MR | 对业务类知识补充 BookStack |
| 代码知识库 Token 过大 | AI 上下文过大、成本高 | Repomix 分模块、摘要化、固定输出路径 |
| RAG 替代人工阅读 | 员工只问 AI 不读规范 | 文档门户为主，AI 问答为辅 |

## 15. 推荐部署形态

### 第一阶段轻量部署

```text
GitLab
  ├─ knowledge-base 仓库
  ├─ GitLab 组/项目权限控制源头内容
  ├─ CI 按权限域构建 Docusaurus/MkDocs
  ├─ 每个权限域独立生成 Pagefind 搜索索引
  └─ 发布到内网 Nginx / OAuth2 Proxy / SSO 网关
```

建议第一阶段采用多站点或多权限域部署：

```text
kb-common.company.local        # 全员可访问
kb-ai-dev.company.local        # 研发可访问
kb-project-mes.company.local   # MES 项目组可访问
kb-project-eap.company.local   # EAP 项目组可访问
kb-security.company.local      # 架构/安全组可访问，默认不进入 RAG
```

优点：

1. 组件少；
2. 运维简单；
3. 授权边界清晰；
4. 静态搜索索引不跨权限域泄露；
5. 上线快；
6. 适合验证知识治理流程。

### 第二阶段增强部署

```text
GitLab
  ↓
CI/CD
  ├─ 构建 Docusaurus
  ├─ 构建 Pagefind
  ├─ 导出 RAG 数据
  └─ 同步 OpenSearch / Qdrant

服务层：
  ├─ Nginx / OAuth2 Proxy / SSO
  ├─ Docusaurus 静态站
  ├─ OpenSearch，全文检索与文档级权限过滤
  ├─ 知识检索 API，统一做权限过滤和审计
  ├─ Haystack / RAGFlow
  ├─ Qdrant / pgvector，内网访问 + 服务账号/API Key
  └─ ai-switch / cx-aicode 接口
```

## 16. 推荐最终技术栈

| 层级 | 推荐 | 备选 | 原因 |
|---|---|---|---|
| 知识源 | Git + Markdown/MDX | BookStack 导出 | 可审计、可版本化、适合 AI 消费 |
| 访问控制 | SSO/OIDC + Nginx/OAuth2 Proxy | GitLab Pages 认证 / 内部门户 | 静态站不承担权限，必须由网关和权限域控制 |
| 人类阅读门户 | Docusaurus | MkDocs | 授权友好，适合多栏目、多版本、多技术栈 |
| 轻量搜索 | Pagefind | Docusaurus 内置搜索插件 | 无后端，运维简单 |
| 企业级搜索 | OpenSearch | Elasticsearch 商业版 | Apache 2.0，适合复杂搜索 |
| 代码知识库 | Repomix | OpenGrok / Zoekt | 适合 AI 编程上下文生成 |
| RAG 框架 | Haystack | RAGFlow | Haystack 更适合自研，RAGFlow 更适合快速落地 |
| 向量库 | Qdrant | pgvector / Milvus | 授权友好，部署和过滤能力较均衡 |
| 在线编辑补充 | BookStack | - | 适合非研发部门维护业务知识 |

---

## 17. 最终建议

建议采用“两主线、一扩展”的建设思路：

### 主线一：人类知识门户

```text
Git + Docusaurus + Pagefind
```

用于解决：

1. 人的阅读；
2. 知识分类；
3. 全文检索；
4. 版本管理；
5. 培训和规范发布。

### 主线二：知识治理流程

```text
Front Matter + MR 审核 + CI 校验 + 过期扫描
```

用于解决：

1. 谁维护；
2. 哪个版本有效；
3. 哪些文档过期；
4. 哪些文档允许 AI 使用；
5. 哪些文档需要进入 RAG。

### 扩展线：AI 编程知识服务

```text
Repomix + /cx-codebase-skill + Haystack/RAGFlow + Qdrant
```

用于解决：

1. AI 编程上下文；
2. 代码库摘要；
3. 规范问答；
4. CodeReview 规则检索；
5. cx-aicode 命令辅助；
6. 项目级知识注入。

---

## 18. 一句话结论

企业知识库要兼顾人的阅读和 AI 使用时，最稳妥的方案不是直接上 AI 知识库平台，而是：

> **以 Docusaurus / MkDocs + Git 作为主知识库，以 Pagefind/OpenSearch 满足人的检索，以 Repomix + Haystack/RAGFlow + Qdrant 满足 AI 编程扩展。**

这条路线的核心优势是：

1. 授权风险低；
2. 人类阅读体验好；
3. 检索能力可逐步增强；
4. 文档可版本化、可审计；
5. 可通过 SSO、分区站点、OpenSearch 权限过滤和 RAG 检索过滤实现授权访问；
6. 与企业 AI 编程流程天然适配；
7. 不会把企业知识治理完全绑定到某个 AI 问答平台。
