# 企业知识库建设方案（Hermes 分层梳理法）

> 版本：v1.0  
> 适用范围：企业内部通用知识库、研发/IT/业务/运维/制度/项目文档知识库  
> 方案定位：以“人可阅读、可检索、可授权、可维护”为主，AI 问答作为可选增强  
> 核心思路：用 Hermes/Agent 分层梳理方法生产结构化知识，用 Docusaurus/MkDocs/BookStack 等平台发布，用 Pagefind/OpenSearch 检索，用统一认证和权限模型保障访问安全。

---

## 1. 背景与目标

企业知识库不能只解决“把文件上传到系统里”的问题，更要解决以下问题：

1. **人能系统阅读**：有清晰目录、主题页、FAQ、术语表、流程图和培训材料。
2. **人能快速检索**：支持全文搜索、按部门/项目/主题/权限过滤。
3. **知识可治理**：有来源、责任人、审核状态、版本、有效期和过期提醒。
4. **访问可授权**：不同部门、项目组、角色看到不同内容。
5. **可持续维护**：知识变更有流程，不能长期堆积过期文档。
6. **可选 AI 问答**：在需要自然语言问答、跨文档总结时，接入 RAG/AI 助手，但不把 RAG 作为知识库本体。

本方案采用：

```text
Hermes/Agent 分层梳理方法
+
Git/Markdown 或 BookStack 作为知识源与发布平台
+
Docusaurus/MkDocs/BookStack 作为人类阅读入口
+
Pagefind/OpenSearch 作为全文检索入口
+
可选 RAGFlow/Haystack + Qdrant/pgvector 作为 AI 问答增强
+
SSO/LDAP/AD/OIDC + 网关 + 索引隔离 + 元数据过滤 作为权限体系
```

## 2. 总体结论

### 2.1 推荐主方案

如果企业内部知识库主要由研发、IT、平台组、架构组、项目组维护，推荐：

```text
Hermes 分层梳理
+
Git + Markdown
+
Docusaurus 或 MkDocs
+
Pagefind / OpenSearch
+
统一认证网关
+
可选 RAG
```

### 2.2 业务协作型补充方案

如果知识库主要由非技术人员维护，例如行政、业务、客服、培训、运营部门，推荐：

```text
Hermes 分层梳理
+
BookStack
+
内置权限/搜索 或 OpenSearch
+
可选 RAG
```

### 2.3 企业统一知识库推荐组合

```text
技术/研发/项目规范类知识：
Git + Markdown + Docusaurus/MkDocs

业务/制度/培训/客服类知识：
BookStack

统一全文检索：
OpenSearch

小规模静态站检索：
Pagefind

AI 问答增强：
RAGFlow 或 Haystack + Qdrant/pgvector

权限：
SSO/LDAP/AD/OIDC + Nginx/OAuth2 Proxy + Git 权限 + 搜索/RAG 权限过滤
```

## 3. 设计原则

### 3.1 人类阅读优先

知识库首先服务于人，而不是服务于 AI。所有正式知识必须能通过目录、主题页、搜索、标签和权限入口被人阅读。

### 3.2 知识源头可审计

正式知识必须有：

- 原始来源；
- 责任人；
- 审核人；
- 审核状态；
- 更新时间；
- 有效期；
- 权限等级；
- 版本记录。

### 3.3 AI 只做整理辅助，不直接发布正式知识

Hermes/Agent 可以生成摘要、FAQ、主题页、目录建议和知识卡片，但不能跳过人工审核直接发布为正式知识。

### 3.4 RAG 是增强层，不是知识库本体

知识库的核心是“可治理的知识资产”，不是向量库。RAG 只用于自然语言问答、跨文档总结、动态知识召回等场景。

### 3.5 权限必须从源头到检索层贯穿

不能只控制网页访问，还要控制：

- 原始文件访问；
- Git 仓库访问；
- 静态站点访问；
- 搜索索引访问；
- RAG 检索访问；
- AI 回答引用范围。

## 5. 分层知识模型

### 5.1 L0 原始资料层

保存所有原始资料，不直接作为正式知识使用。

典型内容：制度文件、项目文档、运维手册、培训材料、会议纪要、Word/PDF/PPT、Markdown、图片、表格、流程图、历史 FAQ、历史问题处理记录。

要求：

1. 保留原始文件。
2. 记录来源。
3. 标注权限。
4. 禁止直接把全部原始资料无差别导入 AI 问答系统。
5. 对敏感资料先分级，再决定是否允许 AI 整理。

### 5.2 L1 文档卡片层

每份原始资料进入知识库前，先生成“文档卡片”。

文档卡片字段：

```yaml
title: 文档标题
summary: 一句话摘要
source_file: 原始文件路径
owner: 责任部门或责任人
reviewer: 审核人
permission: L0/L1/L2/L3/L4/L5
department: 所属部门
project: 所属项目
topic: 知识主题
doc_type: 制度/流程/手册/FAQ/项目文档/培训材料/接口说明
status: draft_ai_generated / pending_review / approved / deprecated / archived
created_at: 创建时间
updated_at: 更新时间
expire_at: 有效期
generated_by: Hermes/Agent/人工
source_hash: 原始文件哈希
```

用途：让每份资料可被管理，方便权限过滤、检索、统计、识别过期文档和 AI 问答引用来源。

### 5.3 L2 主题知识层

Hermes/Agent 将多个文档卡片归纳为主题知识。

典型主题：企业制度、研发规范、运维流程、项目文档、系统操作手册、安全规范、培训资料、FAQ、术语表、常见故障处理、业务流程、审批流程、项目知识地图。

每个主题建议生成：

```text
README.md              主题说明
overview.md            主题概览
faq.md                 常见问题
glossary.md            术语表
process.md             标准流程
related-documents.md   相关文档
change-log.md          变更记录
```

### 5.4 L3 人类阅读层

人类阅读层通过 Docusaurus/MkDocs/BookStack 发布。

要求：

1. 有清晰目录。
2. 有首页导航。
3. 有主题页。
4. 有 FAQ。
5. 有术语表。
6. 有“最后更新时间”和“责任人”。
7. 有“适用范围”和“权限级别”。
8. 重要页面必须有审核状态。
9. 过期页面要有明显提示。
10. 禁止把未经审核的 AI 草稿发布为正式知识。

### 5.5 L4 检索与问答层

检索分两类：

1. **全文检索**：面向人，优先使用 Pagefind 或 OpenSearch。
2. **AI 问答**：面向自然语言问答，按需接入 RAGFlow/Haystack。

全文检索是基础能力，AI 问答是增强能力。

## 6. 组件选型

### 6.1 主知识库平台

| 组件 | 推荐级别 | 授权 | 适用场景 |
|---|---:|---|---|
| Docusaurus | 推荐 | MIT | 技术知识门户、研发规范、项目文档、版本化文档 |
| MkDocs | 推荐 | BSD | 轻量 Markdown 文档站、技术手册、内部规范 |
| BookStack | 推荐补充 | MIT | 业务知识、制度文档、在线编辑、非技术人员维护 |
| Wiki.js | 谨慎 | AGPL-v3 | 功能强，但 AGPL 对企业深度集成更敏感 |
| Dify/FastGPT/MaxKB | 不作为主库 | 许可证/定位需评估 | 更适合 AI 应用/RAG，不适合作为人类主知识库 |

### 6.2 检索组件

| 组件 | 推荐级别 | 授权 | 适用场景 |
|---|---:|---|---|
| Pagefind | 小规模首选 | MIT | 静态站全文搜索，无需后端 |
| OpenSearch | 企业级首选 | Apache 2.0 | 多项目、多权限、统一检索、文档级权限 |
| Elasticsearch | 谨慎 | 许可证需确认 | 企业常见，但授权策略需单独评估 |

### 6.3 AI 问答增强

| 组件 | 推荐级别 | 授权 | 适用场景 |
|---|---:|---|---|
| Haystack | 推荐 | Apache 2.0 | 自研可控 RAG 管线、权限过滤、企业系统集成 |
| RAGFlow | 推荐 | Apache 2.0 | 快速搭建文档问答和 RAG 平台 |
| AnythingLLM | 可选 | MIT | 部门级私有问答入口 |
| Qdrant | 推荐 | Apache 2.0 | 向量检索 |
| pgvector | 推荐 | PostgreSQL License | 已有 PostgreSQL 时低成本接入 |

### 6.4 Hermes/Agent 角色

Hermes/Agent 不作为知识库平台本体，而作为：

- 文档整理助手；
- 知识卡片生成器；
- 主题归纳助手；
- FAQ 生成器；
- 术语表生成器；
- 过期风险检查助手；
- 原始资料到结构化 Markdown 的转换助手；
- 知识维护巡检助手。

## 7. 权限与授权设计

### 7.1 权限等级

| 等级 | 含义 | 示例 |
|---|---|---|
| L0 | 全员可读 | 企业公开制度、通用培训材料 |
| L1 | 部门可读 | 部门流程、部门 FAQ |
| L2 | 项目组可读 | 项目文档、系统说明 |
| L3 | 特定角色可读 | 运维手册、生产系统操作指南 |
| L4 | 敏感知识 | 安全设计、架构细节、客户敏感资料 |
| L5 | 禁止 AI 处理 | 密码、密钥、个人隐私、生产账号、未脱敏客户数据 |

### 7.2 权限控制分层

#### 7.2.1 源头权限

- Git 仓库按目录/项目控制访问。
- BookStack 按书架/书/章节控制访问。
- 原始文件存储按部门/项目授权。

#### 7.2.2 阅读门户权限

Docusaurus/MkDocs 是静态站，不直接承担企业级权限。推荐通过以下方式控制：

- Nginx + LDAP/AD；
- OAuth2 Proxy + OIDC；
- 企业门户 SSO；
- GitLab Pages + 企业登录；
- 内网/VPN 访问限制。

如需项目级隔离，建议拆分站点：

```text
kb-common.company.local      全员知识
kb-rd.company.local          研发知识
kb-it.company.local          IT/运维知识
kb-project-a.company.local   项目 A 知识
kb-security.company.local    安全敏感知识
```

#### 7.2.3 搜索权限

Pagefind 是静态索引，不能把不同权限内容混在一个索引里。使用 Pagefind 时必须：

- 一个权限域一个站点；
- 一个权限域一个搜索索引；
- 不把无权限内容构建进用户可访问的索引。

OpenSearch 适合统一搜索，应启用：用户身份传递、角色过滤、文档级权限、字段级过滤、查询审计。

#### 7.2.4 AI/RAG 权限

AI 问答必须遵守同样权限。检索时必须带过滤条件：

```yaml
user_id: 当前用户
roles: 用户角色
departments: 用户部门
projects: 用户项目
permission_scope:
  - L0
  - L1
  - L2
```

AI 回答只能引用用户有权访问的资料。L5 内容禁止进入 AI 整理、索引和问答流程。

## 9. Hermes 分层整理流程

### 9.1 输入

输入资料包括：原始文档、历史手册、会议纪要、已有 Wiki、项目资料、FAQ、操作流程、培训材料、问题处理记录。

### 9.2 第一步：资料登记

每份资料登记到 `raw/`，并生成基础元数据：文件名、文件类型、来源部门、上传人、上传时间、初始权限、是否允许 AI 处理。

### 9.3 第二步：AI 生成文档卡片

Hermes/Agent 为每份资料生成：标题、摘要、关键点、适用范围、相关部门、权限建议、过期风险、是否需要人工审核、推荐目录位置。

状态为：

```yaml
status: draft_ai_generated
```

### 9.4 第三步：人工初审

知识负责人检查：摘要是否准确、权限是否正确、是否存在敏感信息、是否适合进入知识库、是否需要脱敏、是否需要拆分或合并。

通过后状态变为：

```yaml
status: pending_review
```

### 9.5 第四步：生成主题页

Hermes/Agent 根据多个文档卡片生成：主题概览、流程说明、FAQ、术语表、相关文档清单、新人阅读路径、常见问题处理路径。

### 9.6 第五步：专家审核

专家审核后，状态变为：

```yaml
status: approved
```

只有 `approved` 内容可以进入正式知识库、全文搜索和 AI 问答。

### 9.7 第六步：发布

发布到 Docusaurus/MkDocs、BookStack、OpenSearch/Pagefind、可选 RAG 索引。

### 9.8 第七步：定期巡检

按月或季度检查：是否过期、是否无人负责、是否被频繁搜索但无结果、是否有高频问题未形成知识、是否存在重复文档、是否存在未审核 AI 草稿、是否有敏感文档误入搜索或 RAG。

## 10. 知识生命周期

```text
raw_uploaded
  ↓
draft_ai_generated
  ↓
pending_review
  ↓
approved
  ↓
published
  ↓
needs_update / deprecated / archived
```

| 状态 | 含义 | 是否可发布 | 是否可进入搜索 | 是否可进入 AI 问答 |
|---|---|---:|---:|---:|
| raw_uploaded | 原始资料已上传 | 否 | 否 | 否 |
| draft_ai_generated | AI 生成草稿 | 否 | 否 | 否 |
| pending_review | 等待审核 | 否 | 否 | 否 |
| approved | 审核通过 | 是 | 是 | 可选 |
| published | 已发布 | 是 | 是 | 可选 |
| needs_update | 需要更新 | 可保留但需提示 | 是 | 谨慎 |
| deprecated | 已废弃 | 否或仅历史区 | 否 | 否 |
| archived | 已归档 | 否 | 否 | 否 |

## 12. 检索方案

### 12.1 小规模知识库

适用条件：几百到几千篇文档、权限域较少、主要是静态文档、不需要统一跨部门搜索。

推荐：

```text
Docusaurus/MkDocs + Pagefind
```

要求：每个权限域单独构建，每个站点单独搜索索引，不同权限内容不能混到同一个静态索引。

### 12.2 企业级知识库

适用条件：多部门、多项目、多权限、需要统一搜索入口、需要搜索审计、需要文档级权限过滤。

推荐：

```text
OpenSearch
```

索引元数据建议：

```json
{
  "title": "文档标题",
  "content": "正文内容",
  "owner": "RD",
  "department": "IT",
  "project": "project-a",
  "permission": "L2",
  "status": "approved",
  "tags": ["流程", "FAQ"],
  "updated_at": "2026-07-02",
  "source_url": "https://kb.company.local/..."
}
```

查询时必须带用户权限过滤。

## 13. 可选 AI 问答方案

### 13.1 什么时候需要 AI 问答

以下情况建议引入 RAG/AI 问答：

1. 文档很多，用户不知道该搜什么关键词。
2. 需要自然语言提问。
3. 需要跨文档总结。
4. 需要自动生成答案并附引用来源。
5. 需要客服式知识入口。
6. 需要辅助新人培训。
7. 需要从大量历史问题中找类似案例。

### 13.2 什么时候不需要 AI 问答

以下情况可以先不上 RAG：

1. 文档数量不大。
2. 目录和搜索已经够用。
3. 权限体系尚未稳定。
4. 文档质量还没有治理好。
5. 原始资料里敏感内容较多。
6. 尚未建立审核流程。

### 13.3 AI 问答接入原则

- 只索引 `approved` 内容。
- 不索引 L5 内容。
- 检索时必须按用户权限过滤。
- 回答必须带来源。
- 回答不替代正式制度。
- 高风险问题应返回“请查看原文/联系负责人”。
- 所有问答日志应纳入审计。

- [ ] 确定知识分类体系。
- [ ] 确定权限等级 L0-L5。

任务：

### 阶段 3：知识生命周期管理，2-4 周

目标：避免知识库变成过期文件堆。

任务：

- [ ] 建立用户反馈入口。
- [ ] 建立月度知识巡检机制。
- [ ] 建立重复文档识别机制。
- [ ] 建立热门无结果搜索分析。
- [ ] 建立知识质量评分。

### 阶段 4：可选 AI 问答增强，4-8 周

- [ ] 建立问答审计日志。

## 18. 风险与应对

### 18.1 AI 整理不准确

风险：摘要错误、规则简化、条件遗漏、自行补充不存在的内容。

应对：AI 草稿必须人工审核；每个知识块必须保留来源；正式发布前必须由 Owner 确认；重要知识必须保留原文链接。

### 18.2 权限泄露

风险：静态搜索索引泄露敏感内容；AI 问答召回无权限文档；同一站点混合多权限文档。

应对：按权限域拆站点或拆索引；OpenSearch 启用文档级权限；RAG 检索必须带用户权限过滤；L5 文档禁止进入 AI 和搜索；定期做权限审计。

### 18.3 知识过期

风险：文档长期无人维护；用户看到旧流程；旧版本规范被误用。

应对：每篇文档必须有 `expire_at`；月度过期巡检；过期文档页面显示提醒；无 Owner 文档不得发布。

### 18.4 平台复杂度过高

风险：一开始就引入过多组件；运维成本过大；知识库没上线就陷入平台建设。

应对：第一阶段只做 Git/Docusaurus/Pagefind 或 BookStack；OpenSearch 和 RAG 后置；先做 30-50 篇文档试点；先跑通流程，再扩大范围。

## 19. 推荐实施路线

### 19.1 最推荐路线

```text
第 1 步：确定知识分类和权限分级
第 2 步：建立 Git/Markdown 或 BookStack 知识源
第 3 步：用 Hermes/Agent 生成文档卡片和主题页
第 4 步：人工审核
第 5 步：发布到 Docusaurus/MkDocs/BookStack
第 6 步：接入 Pagefind 或 OpenSearch
第 7 步：建立月度维护机制
第 8 步：知识成熟后再接入 RAG/AI 问答
```

### 19.2 不推荐路线

```text
直接上传大量 PDF/Word 到 RAG
直接让 AI 根据原始资料回答
不做权限分级
不保留来源
不设审核流程
不设 Owner
不设过期检查
```

这种路线短期看起来快，长期会导致知识混乱、误答、越权和不可维护。

## 20. 最终建议

本方案的核心不是选择某一个“知识库软件”，而是建立一套可持续的知识治理流程。

最终推荐：

```text
知识整理方法：
Hermes/Agent 分层梳理

技术型知识库主平台：
Git + Markdown + Docusaurus/MkDocs

业务型知识库补充平台：
BookStack

小规模搜索：
Pagefind

企业级统一搜索：
OpenSearch

可选 AI 问答：
RAGFlow/Haystack + Qdrant/pgvector

权限控制：
SSO/LDAP/AD/OIDC + 网关 + 源头权限 + 搜索/RAG 权限过滤

治理机制：
Owner + Reviewer + 状态流转 + 有效期 + 审计 + 月度巡检
```

一句话总结：

> **Hermes 分层梳理法适合把原始资料整理成结构化知识；Docusaurus/MkDocs/BookStack 适合承载人类阅读；Pagefind/OpenSearch 负责检索；RAG 只作为可选 AI 问答增强；权限和审核必须贯穿整个流程。**
