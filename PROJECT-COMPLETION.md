## 🎓 技术亮点

### 1. 智能检测算法

```bash
# 通过配置文件检测
if [ -f "package.json" ]; then
    if grep -q "react" package.json; then
        frameworks+=("React")
    fi
fi

# 通过文件扫描检测
if find . -maxdepth 3 -name "*.py" | grep -q .; then
    langs+=("Python")
fi
```

```bash

```

```bash

    # 生成 API 相关文档
fi
```

## 🎯 实际应用场景

### 场景 1：新项目启动
**问题**：新项目缺少文档，团队成员难以快速了解项目结构。
**解决**：运行 `./generate-docs-smart.sh` 快速生成初始文档框架。

**问题**：项目从 Vue 迁移到 React，文档需要更新。
**解决**：重新运行脚本，自动识别新技术栈并更新文档。

**问题**：需要详细了解 API 端点和业务流程。
**解决**：运行 `./generate-docs-smart.sh --deep` 生成详细分析报告。

### 场景 4：新成员入职
**问题**：新成员需要快速了解项目架构和业务逻辑。
**解决**：提供自动生成的文档和 PlantUML 图表。

### 场景 5：CI/CD 集成
**问题**：希望文档随代码自动更新。
**解决**：在 CI/CD 流程中集成脚本，自动生成和提交文档。

## 💡 关键决策

### 决策 1：使用 `claude -p` 而非 Claude API

**原因：**
- ✅ 无需 API Key
- ✅ 使用用户已有的 Claude Code
- ✅ 更简单的集成
- ✅ 免费

**权衡：**
- ⚠️ 必须在 Claude Code 环境中运行
- ⚠️ 无法在纯 bash 环境中使用深度分析

### 决策 2：创建独立的 Skill 文件

**原因：**
- ✅ 结构化和可维护
- ✅ 可复用
- ✅ 符合 Claude Code 最佳实践
- ✅ 易于扩展

**权衡：**
- ⚠️ 需要额外的文件
- ⚠️ 用户需要安装 skill

### 决策 3：双模式设计（基础 + 深度）

**原因：**
- ✅ 灵活性
- ✅ 满足不同需求
- ✅ 基础模式快速无依赖
- ✅ 深度模式详细准确

**权衡：**
- ⚠️ 代码复杂度增加
- ⚠️ 需要维护两套逻辑

### 决策 4：JSON 格式的分析报告

**原因：**
- ✅ 结构化
- ✅ 易于解析
- ✅ 可被其他工具使用
- ✅ 标准格式

**权衡：**
- ⚠️ 需要 jq 工具查看
- ⚠️ 不如 Markdown 直观

## 🎓 学到的经验

### 1. 项目分析的复杂性
不同项目的结构差异很大，需要多种检测策略：
- 配置文件检测（package.json, requirements.txt）
- 文件扫描检测（*.js, *.py）
- 目录结构检测（src/, app/）

### 2. AI 辅助代码分析的价值
使用 Claude 进行深度分析可以：
- 理解复杂的业务逻辑
- 提取隐式的架构模式
- 识别代码间的关系

### 3. 文档自动化的重要性
自动生成文档可以：
- 节省大量时间
- 保持文档与代码同步
- 降低文档维护成本
