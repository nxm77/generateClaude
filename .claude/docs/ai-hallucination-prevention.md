# AI 幻觉防护机制

> **版本:** v1.0
> **日期:** 2026-03-22
> **用途:** 防止 AI 编码幻觉导致的错误

---

## 1. 常见幻觉类型

| 类型 | 表现 | 影响 |
|------|------|------|
| **API 幻觉** | 编造不存在的函数/方法 | 编译错误 |
| **语法幻觉** | 错误的语言语法 | 编译/运行错误 |
| **路径幻觉** | 引用不存在的文件路径 | 读写失败 |
| **规范幻觉** | 不符合项目编码规范 | 代码质量问题 |
| **逻辑幻觉** | 遗漏边界条件/异常处理 | 运行时错误 |
| **依赖幻觉** | 引用不存在的库/包 | 构建失败 |

## 2. 防护策略

### 2.1 文件操作防护

**规则:** 操作文件前必须先检查文件是否存在

```
┌─────────────────────────────────────────────────────────┐
│  AI 需要操作文件时                                       │
│       ↓                                                 │
│  1. 使用 Glob 检查文件是否存在                          │
│     └── if (文件不存在) → 提示用户，不执行操作          │
│       ↓                                                 │
│  2. 读取文件验证内容                                    │
│     └── 确认是预期的文件类型/内容                       │
│       ↓                                                 │
│  3. 执行操作                                           │
│       ↓                                                 │
│  4. 操作后验证                                          │
│     └── 检查操作结果是否符合预期                         │
└─────────────────────────────────────────────────────────┘
```

**实现示例:**

```python
# AI 内部检查流程
def safe_write_file(file_path, content):
    # 1. 检查目录是否存在
    dir_path = os.path.dirname(file_path)
    if not os.path.exists(dir_path):
        return f"错误: 目录 {dir_path} 不存在，请先创建"

    # 2. 如果文件已存在，先读取确认
    if os.path.exists(file_path):
        existing = read_file(file_path)
        if not is_same_type(existing, content):
            return f"警告: {file_path} 已存在且类型不匹配"

    # 3. 执行写入
    write_file(file_path, content)
    return f"已写入: {file_path}"
```

### 2.2 API 函数防护

**规则:** 使用 API 前必须查证其存在性

```
┌─────────────────────────────────────────────────────────┐
│  AI 需要使用某个 API/函数                                │
│       ↓                                                 │
│  1. 检查项目 CLAUDE.md 中的 API 说明                    │
│       ↓                                                 │
│  2. 使用 Grep 搜索代码库中的实际使用                    │
│     └── if (搜索结果为空) → 不确定，询问用户            │
│       ↓                                                 │
│  3. 参考实际用法                                        │
│     └── 从现有代码中复制正确的调用模式                  │
│       ↓                                                 │
│  4. 生成代码                                            │
└─────────────────────────────────────────────────────────┘
```

### 2.3 技能引用优先

**规则:** 优先使用已验证的技能文档

```
优先级:
  1. 项目级 CLAUDE.md (已验证的项目规范)
  2. 技术栈 Skills (cpp/vbnet/java 等)
  3. lessons.md (已验证的经验)
  4. instincts/ (自动学习的模式)
  5. 通用知识 (最后才使用)
```

### 2.4 代码验证流程

```
┌─────────────────────────────────────────────────────────┐
│  AI 生成代码后                                           │
│       ↓                                                 │
│  1. 自我审查                                            │
│     ├── 语法检查                                        │
│     ├── API 存在性检查                                  │
│     ├── 文件路径检查                                    │
│     └── 规范符合性检查                                  │
│       ↓                                                 │
│  2. 标记不确定部分                                      │
│     └── 使用 [TODO] 或 [UNCERTAIN] 标记                │
│       ↓                                                 │
│  3. 提示用户验证                                        │
│     └── "请确认以下 API 是否存在: xxx()"               │
│       ↓                                                 │
│  4. 用户确认后执行                                      │
└─────────────────────────────────────────────────────────┘
```

## 3. 实施机制

### 3.1 Pre-Tool-Use Hook

在 AI 执行工具前进行验证：

```json
{
  "hooks": {
    "pre-tool-use": {
      "enabled": true,
      "rules": [
        "write_file: check_directory_exists",
        "edit_file: check_file_exists",
        "read_file: check_file_exists",
        "grep: validate_pattern"
      ]
    }
  }
}
```

### 3.2 AI 行为规则

在 CLAUDE.md 中添加：

```markdown
## AI 编码规则

### 文件操作
- ✅ 使用 Glob 检查文件是否存在
- ✅ 确认路径后再操作
- ❌ 不直接假设文件存在

### API 使用
- ✅ 先搜索代码库确认 API 存在
- ✅ 参考现有代码的使用方式
- ❌ 不编造不存在的 API

### 代码生成
- ✅ 生成后自审语法和规范
- ✅ 标记不确定的内容
- ❌ 不生成未经验证的代码

### 遇到不确定时
- ✅ 使用 AskUserQuestion 询问用户
- ❌ 不盲目假设
```

### 3.3 /verify 命令

新增验证命令：

```bash
# 验证生成的代码
/verify "这段代码使用 xxx() 函数是否正确?"

# 验证 API 存在性
/verify api "std::make_shared"

# 验证文件路径
/verify path "src/core/Device.cpp"
```

## 4. 幻觉检测与纠正

### 4.1 编译时检测

```bash
$ /diagram flow "设备通信流程"

生成过程中...
⚠️ 检测到潜在幻觉:
  - 引用的函数 S1F13_Send() 在代码库中未找到
  - 建议使用: establishCommunication()

请确认:
  [1] 使用建议的函数
  [2] 坚持使用原函数（手动实现）
  [3] 取消
```

### 4.2 运行时检测

```bash
$ git commit

Running pre-commit checks...

⚠️ AI 幻觉检测:
  - 文件: src/Device.cpp
  - 问题: 使用了不存在的宏 INVALID_STATE
  - 建议: 使用 State::DISABLED

是否自动修复?
  [Y]es / [N]o / [A]bort
```

### 4.3 从错误中学习

```bash
# 幻觉纠正后自动记录到 lessons.md
/learn error "使用了不存在的 SECS 函数"

[ERROR] 编造的 SECS 函数
问题: 使用了 S1F13_Send()，实际函数是 establishCommunication()
原因: AI 幻觉，未查证代码库
解决: 使用 Grep 搜索 "establishCommunication" 确认
```
