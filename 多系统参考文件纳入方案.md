# 多系统参考文件纳入方案

## 复核结论

方案 A 可行，推荐采用：每个业务系统作为一个独立的 `skills/business/<system>/` 技能包，系统参考文件放在该技能包内部的 `references/` 目录中，再通过 `flavors/<system>/plugin.json` 组合对应技术栈、业务技能和模板。

该方案的主要优点是：

- 不需要新增顶层 `reference/` 复制机制。
- 业务系统之间天然隔离，避免 MES、EAP、WMS、QMS 等系统知识互相污染。
- 技术栈技能继续复用，业务系统只维护自己的领域知识。
- 初始化目标项目时，业务技能和参考文件可以随 flavor 一起复制到 `.cx/skills/business/<system>/`。

但复核当前仓库实现后，有一个必须处理的初始化问题：当前 `scripts/win/init-project.ps1` 使用 `$componentPathMap` 白名单映射组件路径。新增 `skills/business/<system>` 后，如果没有同步更新映射，`/cx-init --flavor=<system>` 会把该组件识别为 unknown 并跳过。

因此，方案 A 落地前需要二选一：

- 推荐：增强 `init-project.ps1`，允许 `includes` 中真实存在的 `skills/...` 和 `templates/...` 路径直接复制。
- 备选：每新增一个系统，就手工把 `skills/business/<system>`、`templates/docs/<system>` 等路径加入 `$componentPathMap`。

推荐采用第一种，后续新增系统不再需要修改初始化脚本。

## 当前实现约束

当前初始化流程的关键逻辑位于 `scripts/win/init-project.ps1`：

1. 读取 `flavors/<flavor>/plugin.json`。
2. 取出其中的 `includes` 数组。
3. 使用 `$componentPathMap` 将组件名映射为实际插件路径。
4. 将 `skills/...` 复制到目标项目 `.cx/skills/...`。
5. 将 `templates/...` 复制到目标项目 `.cx/templates/...`。

当前复制逻辑本身支持递归复制目录，因此只要 `skills/business/wms` 被识别为有效组件，其内部的 `references/` 会自动复制，不需要额外处理。

例如插件中存在：

```text
skills/business/wms/
├── SKILL.md
└── references/
    ├── architecture.md
    ├── api-contracts.md
    └── business-rules.md
```

初始化后会得到：

```text
.cx/skills/business/wms/
├── SKILL.md
└── references/
    ├── architecture.md
    ├── api-contracts.md
    └── business-rules.md
```

需要注意的隐藏问题：当前 `$componentPathMap` 已包含 `templates/docs/mes` 和 `templates/diagrams/mes`，但没有包含 `templates/docs/eap` 和 `templates/diagrams/eap`，而 `flavors/eap/plugin.json` 中包含了这些路径。按当前实现，EAP 专属模板存在被跳过的风险。这个问题也可以通过“真实路径直接复制”增强一并解决。

另一个配置层问题：当前 `.cx/config.json` 中的 `techStacks` 来自 `-Tech` 参数，而不是自动读取 flavor 的 `techStacks`。如果用户只执行 `/cx-init --flavor=wms`，没有额外传 `--tech=java,vue`，目标项目的 `.cx/config.json` 可能无法准确展示该 flavor 的技术栈。新增多技术栈 flavor 时，建议同步增强初始化脚本：当 `-Tech` 为空且 flavor 中存在 `techStacks` 时，使用 `$flavorConfig.techStacks` 写入 `.cx/config.json`。

## 参考文件拆分规范

不要把一个系统的所有参考材料合成一个巨大的 Markdown 文件。建议按用途拆分，便于按需读取：

| 文件 | 内容 |
|------|------|
| `references/architecture.md` | 系统边界、模块结构、上下游系统、部署关系 |
| `references/api-contracts.md` | API、消息、事件、错误码、接口兼容规则 |
| `references/database-schema.md` | 核心表、字段、索引、关系、历史字段说明 |
| `references/business-rules.md` | 状态机、业务规则、异常流程、校验规则 |
| `references/glossary.md` | 业务术语、缩写、同义词、禁用叫法 |
| `references/legacy-notes.md` | 历史包袱、兼容逻辑、已知风险、迁移注意事项 |

如果某个系统资料较少，可以只保留必要文件，例如：

```text
references/
├── architecture.md
├── business-rules.md
└── glossary.md
```

## `SKILL.md` 编写模板

`SKILL.md` 应作为业务系统入口，不建议塞入大篇参考资料。它应该说明触发条件、系统边界、必读参考文件和硬性业务规则。

示例：`skills/business/wms/SKILL.md`

```markdown
---
name: business-wms
description: "WMS 仓储管理系统业务技能。触发条件：涉及入库、出库、库存、库位、拣货、波次、盘点、补货、库存冻结等 WMS 开发或评审时。"
user-invocable: false
allowed-tools: "Read, Glob, Grep"
metadata:
  version: "2.3.5"
---

# WMS 业务系统技能

## 适用范围

本技能用于 WMS 仓储管理系统相关需求、设计、开发、测试和代码评审。

WMS 负责：

- 入库管理
- 出库管理
- 库存管理
- 库位管理
- 拣货、复核、打包
- 波次策略
- 盘点与库存调整

不属于 WMS 的边界：

- ERP 订单定价与财务结算
- MES 生产执行
- TMS 运输调度

## 必读参考

处理 WMS 需求时，根据任务类型读取以下参考文件：

| 场景 | 参考文件 |
|------|----------|
| 理解系统边界和模块关系 | `references/architecture.md` |
| 修改接口或联调 | `references/api-contracts.md` |
| 修改数据库、字段、查询逻辑 | `references/database-schema.md` |
| 处理业务规则、状态流转、异常流程 | `references/business-rules.md` |
| 确认术语含义 | `references/glossary.md` |

## 工作规则

- 修改库存数量时，必须确认库存流水、可用库存、冻结库存是否同步更新。
- 涉及出库流程时，必须检查订单、波次、拣货、复核、发运状态的一致性。
- 涉及并发扣减库存时，必须确认事务边界和幂等处理。
- 涉及外部系统接口时，必须确认重试、超时、重复消息和补偿逻辑。
```

## 初始化脚本增强建议

推荐对 `scripts/win/init-project.ps1` 做一次通用增强：

- 先查 `$componentPathMap`，兼容已有别名，例如 `core`、`tech/java`、`business/mes`。
- 如果映射不存在，则判断 `includes` 中的值是否为插件内真实存在的 `skills/...` 或 `templates/...` 路径。
- 如果真实存在，则直接复制。
- 如果不存在，则继续输出 warning 并跳过。
- 当 `-Tech` 参数为空且 flavor 中存在 `techStacks` 时，将 `.cx/config.json` 的 `techStacks` 写为 flavor 的 `techStacks`。

建议逻辑：

```powershell
foreach ($component in $components) {
    $sourcePaths = $componentPathMap[$component]

    if ($null -eq $sourcePaths) {
        $normalizedComponent = $component -replace '\\', '/'
        $isAllowedDirectPath = ($normalizedComponent -match '^(skills|templates)/') -and ($normalizedComponent -notmatch '(^|/)\.\.(/|$)')

        if ($isAllowedDirectPath) {
            $candidatePath = Join-Path $PluginRoot ($normalizedComponent -replace '/', '\')
            if (Test-Path $candidatePath) {
                $sourcePaths = @($normalizedComponent)
            }
        }
    }

    if ($null -eq $sourcePaths) {
        Write-Warning "  Unknown component: $component (skipping)"
        continue
    }

    foreach ($src in $sourcePaths) {
        # 复用现有复制逻辑
    }
}
```

`techStacks` 建议逻辑：

```powershell
$resolvedTechStacks = @()
if (-not [string]::IsNullOrWhiteSpace($Tech)) {
    $resolvedTechStacks = $Tech -split ',' | ForEach-Object { $_.Trim() } | Where-Object { $_ }
} elseif ($flavorConfig -and $flavorConfig.techStacks) {
    $resolvedTechStacks = @($flavorConfig.techStacks)
}

$config = @{
    name = $Name
    version = "2.3.5"
    schemaVersion = "2.0.0"
    flavor = $Flavor
    techStacks = $resolvedTechStacks
    components = $components
    # 其余字段沿用现有逻辑
}
```

这段增强后，新增系统只需要新增目录和 flavor，不需要修改 `$componentPathMap`。

如果暂时不改初始化脚本，也可以在 `$componentPathMap` 中显式增加每个系统的路径：

```powershell
"business/wms"              = @("skills/business/wms")
"skills/business/wms"       = @("skills/business/wms")
"templates/docs/wms"        = @("templates/docs/wms")
"templates/diagrams/wms"    = @("templates/diagrams/wms")
```

但这种方式会导致每新增一个系统都要改脚本，不建议作为长期方案。

## 初始化时是否需要额外操作

### 新项目初始化

完成上面的目录和初始化脚本增强后，新项目不需要额外手工复制参考文件。只需要正常使用 flavor 初始化：

```bash
/cx-aicode:cx-init --flavor=wms --name=wms-project
```

或直接调用脚本验证：

```powershell
powershell -ExecutionPolicy Bypass -NoProfile -File .\scripts\win\init-project.ps1 -Name demo-wms -Flavor wms
```

预期结果：

```text
.cx/skills/business/wms/SKILL.md
.cx/skills/business/wms/references/architecture.md
.cx/skills/business/wms/references/api-contracts.md
.cx/skills/business/wms/references/business-rules.md
```

### 已初始化项目

已存在 `.cx/` 的项目需要谨慎处理。当前 `init-project.ps1` 检测到 `.cx/` 后会询问是否覆盖，并把旧目录重命名为备份目录。它不是增量安装组件的工具。

因此对已初始化项目有三种选择：

1. 可以接受重新初始化时：运行 `/cx-init --flavor=<system>`，让脚本备份旧 `.cx/` 后重建。
2. 不希望重建 `.cx/` 时：手工复制 `skills/business/<system>` 到目标项目 `.cx/skills/business/<system>`，并按需更新 `.cx/config.json` 的 `components` 和 `flavor` 字段。
3. 后续增强：增加一个专门的增量安装命令，例如 `/cx-update --with=skills/business/wms`，但这不属于方案 A 的最小落地范围。

### 插件自身更新后

如果用户使用的是已安装的 Claude Code 插件缓存，而不是直接从当前源码目录运行，修改插件源码后需要重新安装或更新插件，使新 flavor 和新 skill 进入插件缓存。

本仓库开发验证可以使用：

```bash
claude plugin marketplace add ./
claude plugin install cx-aicode
```

如果插件管理器支持覆盖安装或更新，应按实际发布流程执行。

## 验收清单

新增一个系统后，至少检查以下内容：

1. `flavors/<system>/plugin.json` 存在，JSON 语法有效。
2. `skills/business/<system>/SKILL.md` 存在，frontmatter 包含 `name`、`description`、`user-invocable`、`allowed-tools`、`metadata.version`。
3. `skills/business/<system>/references/` 中的参考文件按用途拆分。
4. `flavors/<system>/plugin.json` 的 `includes` 包含所需技术栈、质量规则、测试规则和业务 skill。
5. 初始化脚本已支持直接复制真实存在的 `skills/...` 和 `templates/...` 路径，或 `$componentPathMap` 已显式包含新增系统路径。
6. 执行初始化后，目标项目存在 `.cx/skills/business/<system>/references/`。
7. 若系统有专属模板，目标项目存在 `.cx/templates/docs/<system>/` 或 `.cx/templates/diagrams/<system>/`。
8. 若修改了初始化脚本，运行 PowerShell 回归测试。

建议验证命令：

```powershell
powershell -ExecutionPolicy Bypass -NoProfile -File .\scripts\win\init-project.ps1 -Name demo-wms -Flavor wms
Test-Path .\.cx\skills\business\wms\references\architecture.md
powershell -File .\tests\powershell\run_tests.ps1
```

如果只新增文档、skill 和 flavor，没有改脚本，则至少执行一次初始化验证，确认没有出现 `Unknown component` warning。

## 维护规则

长期维护建议固定以下规则：

- 一个业务系统一个 `skills/business/<system>/`。
- 一个业务系统一个 `flavors/<system>/plugin.json`。
- 业务系统参考资料只放在 `skills/business/<system>/references/`。
- 技术栈规范继续放在 `skills/tech/*`，不要复制到业务系统目录中。
- 质量规则继续放在 `skills/quality/*`。
- 测试规则继续放在 `skills/test/*`。
- 系统专属文档模板放在 `templates/docs/<system>/`。
- 系统专属图表模板放在 `templates/diagrams/<system>/`。
- 多系统共享的规则不要放进某个系统目录，应抽到通用技术、质量或流程技能中。
- `SKILL.md` 只写入口、边界、索引和硬规则，详细资料放 `references/`。

## 推荐落地顺序

1. 选择一个系统作为样板，例如 `wms`。
2. 创建 `skills/business/wms/SKILL.md`。
3. 将系统参考文件拆分到 `skills/business/wms/references/`。
4. 如有必要，创建 `patterns.md`、`debugging.md`、`coding-standards.md`。
5. 创建 `flavors/wms/plugin.json`，组合 Java、Vue、业务技能和模板。
6. 增强 `scripts/win/init-project.ps1`，支持真实存在的 `skills/...` 和 `templates/...` 路径直接复制。
7. 执行初始化验证，确认 `.cx/skills/business/wms/references/` 完整复制。
8. 补充或调整 `README.md`、`commands/cx-init.md` 中的 flavor 示例。
9. 按相同模式扩展其他系统。

## 最终建议

方案 A 可以作为当前插件的标准多系统接入方式。最小可落地改动是：新增业务 skill 包、新增系统 flavor、增强初始化脚本的路径识别逻辑。完成后，用户初始化新项目时只需要选择对应 flavor，不需要额外手工复制参考文件。

对已有项目不要默认重新初始化，除非用户明确接受 `.cx/` 备份重建。已有项目更适合手工补充业务 skill，或后续提供一个真正的增量安装命令。
