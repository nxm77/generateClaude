# CX-Aicode 统计功能操作手册

**文档版本**：v2.3.9  
**更新日期**：2026-05-18  
**适用对象**：平台运维、插件管理员、业务项目负责人、审计统计人员

---

## 1. 功能范围

本文说明如何启用并查看 cx-aicode 当前已实现的两类统计能力：

1. **插件技能统计**：按关键 `/cx-*` 流程命令调用统计。当前代码采集的是 `/cx-spec`、`/cx-brainstorm`、`/cx-work`、`/cx-codereview`、`/cx-verify`、`/cx-test`、`/cx-debug`，不是逐个 `skills/*/SKILL.md` 文件的真实加载次数。
2. **AI 编码行数统计**：记录 AI 命令产生的 patch、提交前 staged diff、commit 绑定关系，并统计 AI / human / unknown 行数。

两类统计都写入业务项目的 `.cx/usage/`。默认只本地落盘；集中上报需要显式启用 `central-upload` 并部署 `server/usage-api`。

---

## 2. 总体链路

```text
Claude Code 用户输入
  -> hooks/hooks.json
  -> hooks/dispatch-hook.js
  -> hooks/usage/command-usage.js
  -> .cx/usage/cx-command-events.jsonl
  -> 可选上传到 usage API: cx_command_usage_events

AI 修改代码
  -> UserPromptSubmit: ai-stats-cli command-start
  -> Stop: ai-stats-cli command-end
  -> .cx/usage/ai-patch-events.jsonl + patches/

Git 提交
  -> pre-commit: ai-stats-cli pre-commit
  -> commit-msg: 写入 Spec-Id / AI-Stats-Id trailer
  -> post-commit: 绑定 commitSha
  -> pre-push: 上传 staged / commit / patch 统计
  -> usage API: cx_ai_staged_stats / cx_ai_patch_events
```

默认模式不会访问网络；上传失败不会阻塞 Claude Code，也不会阻塞 Git 提交流程中主业务动作。

## 3. 启用前检查

### 3.1 客户端要求

业务项目机器需要：

| 项目 | 要求 | 验证命令 |
|---|---|---|
| Claude Code | 已安装并能加载 cx-aicode 插件 | `claude plugin list` |
| Node.js | 18+ | `node --version` |
| Git | 需要统计 AI 编码行数时必须是 Git 仓库 | `git status` |
| cx-aicode 初始化 | 业务项目存在 `.cx/config.json` | 检查 `.cx/` |

初始化业务项目通常使用：

```powershell
cd D:\your-business-project
powershell -NoProfile -ExecutionPolicy Bypass `
  -File D:\cx-aicode\scripts\win\init-project.ps1 `
  -Name your-project `
  -Flavor mes
```

初始化脚本会生成：

```text
.cx/config.json
.cx/usage/
.cx/hooks/
.claude/settings.json
```

如果当前目录是 Git 仓库，初始化脚本会复制 Git hooks 到 `.cx/hooks/`，并在没有已有 hooksPath 时执行：

```bash
git config core.hooksPath .cx/hooks
```

如果项目已经配置了其他 `core.hooksPath`，脚本不会覆盖，需要把下面四个调用合并到既有 Git hook 中：

```bash
node "<插件根>/hooks/usage/ai-stats-cli.js" pre-commit
node "<插件根>/hooks/usage/ai-stats-cli.js" commit-msg "$1"
node "<插件根>/hooks/usage/ai-stats-cli.js" post-commit
node "<插件根>/hooks/usage/ai-stats-cli.js" pre-push
```

Windows 项目也使用这些 shell hook 文件，Git for Windows 会执行 `.cx/hooks/*`。

### 3.2 服务端要求

集中上报服务器需要：

| 项目 | 要求 |
|---|---|
| Node.js | 20+ |
| 数据库 | 默认 MySQL 8；可选 PostgreSQL 13+ 或 Oracle |
| 网络 | 客户端能访问 usage API 地址 |
| Token | 服务端和客户端 Bearer Token 一致 |

服务端源码目录：

```text
server/usage-api/
```

## 4. 服务端部署

### 4.1 MySQL 8 部署

在服务器上进入 usage API 目录：

```bash
cd /opt/cx-aicode/server/usage-api
npm ci
```

创建数据库和账号后执行迁移。示例：

```bash
mysql --host=mysql-host --user=cx_usage --password --database=cx_usage \
  < migrations/001_usage_events.mysql.sql
```

设置环境变量并启动：

```bash
export DB_DIALECT=mysql
export DATABASE_URL="mysql://cx_usage:password@mysql-host:3306/cx_usage"
export USAGE_API_TOKENS="dept-a-prod:plain-token-value"
export PORT=3000
export USAGE_RATE_LIMIT_MAX=600
npm start
```

PowerShell 示例：

```powershell
cd server\usage-api
npm ci

$env:DB_DIALECT = "mysql"
$env:DATABASE_URL = "mysql://cx_usage:password@mysql-host:3306/cx_usage"
$env:USAGE_API_TOKENS = "dept-a-prod:plain-token-value"
$env:PORT = "3000"
$env:USAGE_RATE_LIMIT_MAX = "600"

Get-Content .\migrations\001_usage_events.mysql.sql -Raw |
  mysql --host=mysql-host --user=cx_usage --password --database=cx_usage

npm start
```

### 4.2 PostgreSQL 部署

```bash
cd /opt/cx-aicode/server/usage-api
npm ci

export DB_DIALECT=postgres
export DATABASE_URL="postgres://cx_usage:password@postgres-host:5432/cx_usage"
export USAGE_API_TOKENS="dept-a-prod:plain-token-value"

psql "$DATABASE_URL" -f migrations/001_usage_events.sql
npm start
```

### 4.3 Oracle 部署

```bash
cd /opt/cx-aicode/server/usage-api
npm ci

export DB_DIALECT=oracle
export ORACLE_USER=cx_usage
export ORACLE_PASSWORD=password
export ORACLE_CONNECT_STRING="oracle-host:1521/service"
export USAGE_API_TOKENS="dept-a-prod:plain-token-value"

sqlplus "$ORACLE_USER/$ORACLE_PASSWORD@$ORACLE_CONNECT_STRING" \
  @migrations/001_usage_events.oracle.sql
npm start
```

Oracle 模式使用 `oracledb` optional dependency，部署机需要满足 Oracle Node 驱动运行条件。

### 4.4 服务端环境变量

| 变量 | 必填 | 默认值 | 说明 |
|---|---:|---|---|
| `DB_DIALECT` | 否 | `mysql` | `mysql`、`postgres` / `postgresql`、`oracle` |
| `DATABASE_URL` | MySQL/PostgreSQL 必填 | 无 | 数据库连接字符串 |
| `ORACLE_USER` | Oracle 必填 | 无 | Oracle 用户 |
| `ORACLE_PASSWORD` | Oracle 必填 | 无 | Oracle 密码 |
| `ORACLE_CONNECT_STRING` | Oracle 必填 | 无 | Oracle 连接串 |
| `USAGE_API_TOKENS` | 是 | 无 | Bearer token 列表 |
| `PORT` | 否 | `3000` | HTTP 端口 |
| `HOST` | 否 | `0.0.0.0` | 监听地址 |
| `USAGE_RATE_LIMIT_MAX` | 否 | `600` | 每分钟每 token/IP 请求上限 |
| `USAGE_MAX_BATCH_SIZE` | 否 | `100` | 单批最大事件数 |
| `USAGE_BODY_LIMIT` | 否 | `262144` | 请求 body 最大字节数 |

`USAGE_API_TOKENS` 支持命名 token：

```text
dept-a-prod:plain-token-value,dept-b-test:another-token
```

数据库中会记录 `token_name`，便于按团队、环境或项目组拆分统计。

### 4.5 健康检查

```bash
curl http://usage-api-host:3000/health
curl http://usage-api-host:3000/ready
```

含义：

| 接口 | 含义 |
|---|---|
| `/health` | 进程存活，不要求数据库连通 |
| `/ready` | 数据库就绪检查；数据库不可用时返回 503 |

### 4.6 服务端表

MySQL 迁移会创建：

| 表 | 用途 |
|---|---|
| `cx_command_usage_events` | `/cx-*` 命令调用事件 |
| `cx_ai_staged_stats` | AI 行数统计的 staged / bound / uploaded 结果 |
| `cx_ai_patch_events` | AI 命令 patch 事件审计 |
| `cx_ai_stats_audit_events` | statsId / commitSha 绑定冲突等审计异常 |

常用索引：

| 索引 | 用途 |
|---|---|
| `(spec_id, command_name)` | 按需求和命令统计覆盖率 |
| `(project_id, event_time)` | 按项目时间线统计 |
| `(git_user_email, event_time)` | 按用户统计 |
| `(event_time)` | 按时间范围统计或归档 |

### 4.7 部署前回归

服务端上线前运行：

```bash
cd server/usage-api
npm test
npm audit --json
```

当前代码中服务端路由使用 `items` 作为上传数组字段，校验函数处存在批量字段口径需要回归确认的风险。若 `/api/v1/cx-aicode/*` 批量接口返回 `400 body.events must be an array`，先运行服务端测试并修复 schema / route 的字段一致性，再推广 `central-upload`。

## 5. 客户端配置

客户端配置文件位于业务项目：

```text
.cx/config.json
```

### 5.1 默认本地统计

初始化后默认是本地模式：

```json
{
  "usageAudit": {
    "enabled": true,
    "mode": "local-only",
    "upload": {
      "enabled": false,
      "endpointEnv": "CX_USAGE_UPLOAD_ENDPOINT",
      "tokenEnv": "CX_USAGE_UPLOAD_TOKEN",
      "builtinTokenEnv": "CX_USAGE_BUILTIN_TOKEN"
    }
  },
  "aiStats": {
    "enabled": true
  }
}
```

`local-only` 行为：

- 采集 `/cx-*` 命令事件。
- 采集 AI patch 和 staged stats。
- 只写 `.cx/usage/`。
- 不生成命令统计 pending 队列。
- 不访问服务端。

### 5.2 启用集中上报

将业务项目 `.cx/config.json` 改为：

```json
{
  "usageAudit": {
    "enabled": true,
    "mode": "central-upload",
    "upload": {
      "enabled": true,
      "endpoint": "https://usage-api.company.com",
      "endpointEnv": "CX_USAGE_UPLOAD_ENDPOINT",
      "tokenEnv": "CX_USAGE_UPLOAD_TOKEN",
      "builtinTokenEnv": "CX_USAGE_BUILTIN_TOKEN",
      "batchSize": 50,
      "maxBatchSize": 100,
      "timeoutMs": 800,
      "maxRunMs": 2000,
      "minIntervalMs": 60000,
      "initialBackoffMs": 60000,
      "maxBackoffMs": 3600000,
      "jitterRatio": 0.2,
      "lockTtlMs": 30000
    }
  }
}
```

客户端会在 endpoint 后自动拼接具体 API 路径：

| 数据 | 自动请求路径 |
|---|---|
| 命令调用事件 | `/api/v1/cx-aicode/events/batch` |
| AI patch 事件 | `/api/v1/cx-aicode/ai-patches` |
| staged stats | `/api/v1/cx-aicode/staged-stats` |
| staged stats 绑定 | `/api/v1/cx-aicode/staged-stats/:statsId/bind` |
| commit stats | `/api/v1/cx-aicode/commits` |

因此推荐把 `endpoint` 配成服务根地址：

```json
"endpoint": "https://usage-api.company.com"
```

不要把 path 配成最终 API 地址，避免重复拼接路径。

### 5.3 通过环境变量覆盖上传地址

如果多个项目使用统一服务端，可以不在项目内写死 endpoint，只设置环境变量：

```powershell
$env:CX_USAGE_UPLOAD_ENDPOINT = "https://usage-api.company.com"
$env:CX_USAGE_UPLOAD_TOKEN = "plain-token-value"
```

Bash：

```bash
export CX_USAGE_UPLOAD_ENDPOINT="https://usage-api.company.com"
export CX_USAGE_UPLOAD_TOKEN="plain-token-value"
```

地址优先级：

1. `.cx/config.json` 的 `usageAudit.upload.endpoint`
2. `usageAudit.upload.endpointEnv` 指定的环境变量，默认 `CX_USAGE_UPLOAD_ENDPOINT`

Token 优先级：

1. `usageAudit.upload.tokenEnv` 指定的环境变量，默认 `CX_USAGE_UPLOAD_TOKEN`
2. `usageAudit.upload.builtinTokenEnv` 指定的环境变量，默认 `CX_USAGE_BUILTIN_TOKEN`

不要把 token 明文写入 `.cx/config.json` 或仓库文件。

### 5.4 关闭统计

完全关闭两类 usage 统计中的命令调用采集：

```json
{
  "usageAudit": {
    "mode": "disabled"
  }
}
```

关闭 AI 编码行数统计：

```json
{
  "aiStats": {
    "enabled": false
  }
}
```

如果只想本地保留、不上传，使用：

```json
{
  "usageAudit": {
    "mode": "local-only"
  }
}
```

### 5.5 SPEC-ID 解析配置

命令统计会尝试从用户输入和项目状态中解析 SPEC-ID。默认正则：

```json
{
  "usageAudit": {
    "specId": {
      "pattern": "\\b[A-Z][A-Z0-9]{1,15}-[A-Z0-9][A-Z0-9_-]{1,31}\\b",
      "prefixWhitelist": []
    },
    "requireSpecId": false,
    "warnWhenUnbound": true,
    "saveRawPrompt": false
  }
}
```

建议：

- 需求 ID 使用 `MES-0535`、`OPT-0123` 这类稳定编号。
- 不启用 `saveRawPrompt`，默认不会保存 prompt 正文。
- `requireSpecId=false` 时无法识别的事件会归到 `UNBOUND`。

### 5.6 AI 行数统计配置

默认启用：

```json
{
  "aiStats": {
    "enabled": true,
    "include": [
      "**/*.java",
      "**/*.vue",
      "**/*.js",
      "**/*.ts",
      "**/*.tsx",
      "**/*.jsx",
      "**/*.cpp",
      "**/*.c",
      "**/*.h",
      "**/*.hpp",
      "**/*.cs",
      "**/*.py",
      "**/*.sql",
      "**/*.xml",
      "**/*.json",
      "**/*.md"
    ],
    "exclude": [
      "**/node_modules/**",
      "**/dist/**",
      "**/build/**",
      "**/target/**",
      "**/vendor/**",
      "**/*.min.js",
      "**/*.lock",
      "**/package-lock.json",
      "**/yarn.lock",
      "**/pnpm-lock.yaml"
    ]
  }
}
```

只统计 `include` 且不匹配 `exclude` 的 staged 文件。锁文件、构建产物、依赖目录默认排除。

### 5.7 dirty workspace 策略

推荐使用新配置：

```json
{
  "aiStats": {
    "workspace": {
      "dirtyWorkspacePolicy": "allow_with_low_confidence",
      "dirtyWorkspaceConfidence": "LOW"
    }
  }
}
```

可选值：

| 值 | 行为 |
|---|---|
| `allow_with_low_confidence` | 允许统计，但置信度标为 `LOW`，默认值 |
| `allow_with_warning` | 允许统计，保留低置信度警示 |
| `deny` | 脏工作区时不信任统计，结果低置信度 |

脏工作区指 AI 命令开始前已有未提交 diff；此时无法完全区分 AI 和人工已有修改。

### 5.8 恢复、格式化和清理策略

```json
{
  "aiStats": {
    "recovery": {
      "enabled": true,
      "recoverBeforeAnalyzeStaged": true,
      "pendingEventTtlHours": 24,
      "recoveredEventConfidence": "MEDIUM",
      "dirtyWorkspaceRecoveredConfidence": "LOW",
      "failedRecoveryPolicy": "mark_unknown"
    },
    "format": {
      "enabled": true,
      "detectFormatOnly": true,
      "patterns": [
        "[format]",
        "chore(format)",
        "format only",
        "chore: format",
        "style(format)"
      ],
      "diffWhitespaceMode": true,
      "normalizedChangeRatioThreshold": 0.1,
      "largeFormatFileThreshold": 20,
      "formatExactConfidence": "HIGH",
      "formatSuspectedConfidence": "MEDIUM"
    },
    "cleanup": {
      "keepConsumedEventsDays": 7,
      "keepUploadedStatsDays": 30,
      "keepStaleEventsDays": 7,
      "keepPatchFilesDays": 30
    }
  }
}
```

说明：

- `recovery` 用于恢复会话中断或 Stop hook 未完成的 AI patch 事件。
- `format` 用于识别纯格式化提交，降低误算。
- `cleanup` 控制本地 `.cx/usage/` 中已消费事件、过期 patch 文件和已上传 stats 的保留时间。

## 6. 采集触发规则

### 6.1 插件命令统计

采集入口：

```text
hooks/dispatch-hook.js
```

触发条件：

- Hook 事件为 `UserPromptSubmit`。
- 用户输入中包含目标命令。
- 当前目录存在 `.cx/`。
- `usageAudit.mode` 不是 `disabled`。

采集命令：

| 命令 | 说明 |
|---|---|
| `cx-spec` | 规范定稿 |
| `cx-brainstorm` | 需求探索 |
| `cx-work` | 执行实现 |
| `cx-codereview` | 代码审查 |
| `cx-verify` | 验证门禁 |
| `cx-test` | 测试生成 / 测试流程 |
| `cx-debug` | 调试流程 |

本地事件文件：

```text
.cx/usage/cx-command-events.jsonl
```

`central-upload` 时额外写入：

```text
.cx/usage/pending-events.jsonl
.cx/usage/acked-events.jsonl
.cx/usage/upload-backoff.json
.cx/usage/last-flush.json
.cx/usage/flush.lock
```

### 6.2 命令统计上传

以下 Hook 会尝试后台 flush：

| Hook | 行为 |
|---|---|
| `SessionStart` | 会话启动时尝试上传 pending |
| `UserPromptSubmit` | 本次有采集事件时尝试上传 |
| `Stop` | 会话结束时尝试上传 |
| `StopFailure` | 会话异常结束时尝试上传 |

上传失败时：

- 事件继续保留在 `pending-events.jsonl`。
- 写入或更新 `upload-backoff.json`。
- 不影响 Claude Code hook stdout。
- 不改变命令允许 / 拒绝判断。

### 6.3 AI 编码行数统计

AI 行数统计由 Claude hooks 和 Git hooks 共同完成：

| 阶段 | 触发 | 写入内容 |
|---|---|---|
| AI 命令开始 | `UserPromptSubmit` 调用 `command-start` | 保存 AI 前 diff、创建 `patchEventId` |
| AI 命令结束 | `Stop` 调用 `command-end` | 生成增量 patch 和 `aiPatchLines` |
| 提交前 | `pre-commit` | 分析 staged diff，生成 `statsId` |
| 提交信息 | `commit-msg` | 写入 `Spec-Id`、`AI-Stats-Id`、`AI-Stats-Version` |
| 提交后 | `post-commit` | 绑定 `commitSha` |
| 推送前 | `pre-push` | 上传 staged stats、bind、commit stats、patch audit |

AI 统计本地文件：

```text
.cx/usage/ai-patch-events.jsonl
.cx/usage/staged-stats-events.jsonl
.cx/usage/cache/ai-patch-state.json
.cx/usage/cache/staged-stats-state.json
.cx/usage/patches/
.cx/usage/active-ai-commands/
```

提交信息中会追加：

```text
Spec-Id: MES-0535
AI-Stats-Id: st_...
AI-Stats-Version: 1
```

不要手动删除这些 trailer；它们用于 post-commit 和 pre-push 绑定统计。

## 7. 本地查看数据

### 7.1 查看命令统计流水

PowerShell：

```powershell
Get-Content .\.cx\usage\cx-command-events.jsonl -Tail 20
```

Bash：

```bash
tail -n 20 .cx/usage/cx-command-events.jsonl
```

每行是一条 JSON 事件，常见字段：

| 字段 | 说明 |
|---|---|
| `eventId` | 事件唯一 ID |
| `eventType` | `cx_command_invoked` |
| `timestamp` | 事件时间 |
| `command` | 命令名 |
| `specId` | 需求编号；无法识别时为 `UNBOUND` |
| `specIdSource` | SPEC-ID 来源 |
| `projectId` | 项目路径哈希 |
| `gitUserName` | Git 用户名 |
| `gitUserEmail` | Git 邮箱 |
| `sessionId` | Claude 会话 ID |
| `pluginVersion` | 插件版本 |

### 7.2 生成命令统计报表

在 cx-aicode 插件源码目录运行：

```bash
node scripts/usage-summary.js <业务项目目录>
```

示例：

```powershell
cd D:\cx-aicode
node scripts\usage-summary.js D:\your-business-project
```

输出文件：

```text
<业务项目>/.cx/usage/reports/cx-command-usage-summary.json
<业务项目>/.cx/usage/reports/cx-command-usage-summary.md
<业务项目>/.cx/usage/reports/cx-command-usage-summary.csv
```

报表按 SPEC-ID 汇总：

| 字段 | 说明 |
|---|---|
| `fullyCompliant` | 是否覆盖完整流程 |
| `coverageRate` | 流程覆盖率 |
| `missingCommands` | 缺失阶段 |
| `commands` | 每个命令调用次数 |

完整度口径：

```text
需求阶段：cx-spec 或 cx-brainstorm 二选一
实现阶段：cx-work
测试阶段：cx-test
审查阶段：cx-codereview
验证阶段：cx-verify
```

`cx-debug` 会被采集到流水和服务端，但本地完整度报表当前不把它计入必选阶段。

### 7.3 查看 AI 统计状态

在业务项目根目录运行：

```bash

```

```powershell

```

输出包括：

```text
AI Stats: enabled
Pending patch events: 0
Patched events: 1
Consumed events: 3
Stale events: 0
Failed events: 0

Uploaded stats: 2
```

含义：

| 项 | 含义 |
|---|---|

| `Consumed events` | 已被某次提交统计消费 |
| `Stale events` | 过期或无法恢复的事件 |
| `Failed events` | 恢复或分析失败事件 |
| `Unbound staged stats` | 已分析 staged diff，但还没绑定 commit |
| `Bound staged stats` | 已绑定 commit，等待上传或本地标记完成 |
| `Uploaded stats` | 已上传或在 local-only 模式下已标记完成 |

## 8. 服务端查看数据

以下 SQL 以 MySQL 8 为例。PostgreSQL / Oracle 需要替换 JSON 函数语法。

### 8.1 查看命令调用总量

```sql
SELECT
  command_name,
  COUNT(*) AS calls
FROM cx_command_usage_events
WHERE event_time >= CURRENT_DATE - INTERVAL 30 DAY
GROUP BY command_name
ORDER BY calls DESC;
```

### 8.2 按 SPEC-ID 查看流程覆盖

```sql
SELECT
  spec_id,
  SUM(command_name = 'cx-spec') AS cx_spec,
  SUM(command_name = 'cx-brainstorm') AS cx_brainstorm,
  SUM(command_name = 'cx-work') AS cx_work,
  SUM(command_name = 'cx-test') AS cx_test,
  SUM(command_name = 'cx-codereview') AS cx_codereview,
  SUM(command_name = 'cx-verify') AS cx_verify,
  SUM(command_name = 'cx-debug') AS cx_debug
FROM cx_command_usage_events
WHERE spec_id <> 'UNBOUND'
GROUP BY spec_id
ORDER BY spec_id;
```

完整流程可用以下条件筛选：

```sql
SELECT
  spec_id
FROM cx_command_usage_events
GROUP BY spec_id
HAVING
  SUM(command_name IN ('cx-spec', 'cx-brainstorm')) > 0
  AND SUM(command_name = 'cx-work') > 0
  AND SUM(command_name = 'cx-test') > 0
  AND SUM(command_name = 'cx-codereview') > 0
  AND SUM(command_name = 'cx-verify') > 0;
```

### 8.3 按用户查看命令调用

```sql
SELECT
  git_user_email,
  command_name,
  COUNT(*) AS calls
FROM cx_command_usage_events
WHERE event_time >= CURRENT_DATE - INTERVAL 30 DAY
GROUP BY git_user_email, command_name
ORDER BY git_user_email, calls DESC;
```

### 8.4 查看未绑定 SPEC-ID

```sql
SELECT
  git_user_email,
  command_name,
  COUNT(*) AS calls
FROM cx_command_usage_events
WHERE spec_id = 'UNBOUND'
GROUP BY git_user_email, command_name
ORDER BY calls DESC;
```

如果 `UNBOUND` 过多，优先检查团队是否按 `/cx-work --spec=<SPEC-ID>` 使用命令，或 `.cx/status/index.json` 是否有有效 `activeChange`。

### 8.5 查看 AI 总行数

```sql
SELECT
  spec_id,
  SUM(COALESCE(JSON_EXTRACT(raw_stats, '$.summary.aiLines.added'), 0)) AS ai_added,
  SUM(COALESCE(JSON_EXTRACT(raw_stats, '$.summary.aiLines.deleted'), 0)) AS ai_deleted,
  SUM(COALESCE(JSON_EXTRACT(raw_stats, '$.summary.aiLines.modified'), 0)) AS ai_modified,
  SUM(COALESCE(JSON_EXTRACT(raw_stats, '$.summary.humanLines.added'), 0)) AS human_added,
  SUM(COALESCE(JSON_EXTRACT(raw_stats, '$.summary.humanLines.deleted'), 0)) AS human_deleted,
  SUM(COALESCE(JSON_EXTRACT(raw_stats, '$.summary.humanLines.modified'), 0)) AS human_modified
FROM cx_ai_staged_stats
WHERE status IN ('BOUND', 'UPLOADED')
GROUP BY spec_id
ORDER BY spec_id;
```

MySQL 返回 JSON 类型时，可显式转换：

```sql
CAST(JSON_UNQUOTE(JSON_EXTRACT(raw_stats, '$.summary.aiLines.added')) AS UNSIGNED)
```

### 8.6 按用户查看 AI 行数

`gitUserEmail` 存在于 `raw_stats` 中：

```sql
SELECT
  JSON_UNQUOTE(JSON_EXTRACT(raw_stats, '$.gitUserEmail')) AS git_user_email,
  SUM(CAST(JSON_UNQUOTE(JSON_EXTRACT(raw_stats, '$.summary.aiLines.added')) AS UNSIGNED)) AS ai_added,
  SUM(CAST(JSON_UNQUOTE(JSON_EXTRACT(raw_stats, '$.summary.aiLines.deleted')) AS UNSIGNED)) AS ai_deleted,
  SUM(CAST(JSON_UNQUOTE(JSON_EXTRACT(raw_stats, '$.summary.aiLines.modified')) AS UNSIGNED)) AS ai_modified
FROM cx_ai_staged_stats
WHERE status IN ('BOUND', 'UPLOADED')
GROUP BY JSON_UNQUOTE(JSON_EXTRACT(raw_stats, '$.gitUserEmail'))
ORDER BY ai_added DESC;
```

### 8.7 查看低置信度统计

```sql
SELECT
  spec_id,
  commit_sha,
  JSON_UNQUOTE(JSON_EXTRACT(raw_stats, '$.confidence')) AS confidence,
  JSON_EXTRACT(raw_stats, '$.summary') AS summary,
  updated_at
FROM cx_ai_staged_stats
WHERE JSON_UNQUOTE(JSON_EXTRACT(raw_stats, '$.confidence')) IN ('LOW', 'MEDIUM')
ORDER BY updated_at DESC;
```

常见原因：

- AI 命令开始前已有未提交 diff。
- 多个 AI patch 事件修改同一 staged 文件。
- staged 文件只消费了部分 AI patch。
- 会话中断后通过 recovery 恢复。
- merge / cherry-pick 场景无法精确归因。

## 10. 排障

### 10.1 `.cx/usage/pending-events.jsonl` 持续增长

可能原因：

- `usageAudit.mode=central-upload` 已启用，但服务端不可达。
- `CX_USAGE_UPLOAD_TOKEN` 未设置或与服务端不一致。
- endpoint 写成 `unconfigured.*` 或无效 URL。
- 服务端返回 401、413、429、503。
- 服务端批量字段 schema 与客户端上传 body 不一致。

处理：

```bash
curl http://usage-api-host:3000/health
curl http://usage-api-host:3000/ready
```

检查客户端：

```powershell
echo $env:CX_USAGE_UPLOAD_ENDPOINT
echo $env:CX_USAGE_UPLOAD_TOKEN
Get-Content .\.cx\config.json
Get-Content .\.cx\usage\usage-errors.log -Tail 20
```

修复后手动 flush：

```powershell
node D:\cx-aicode\hooks\usage\command-usage.js flush --force
```

### 10.2 默认模式没有 pending 文件

这是正常行为。`local-only` 只写：

```text
.cx/usage/cx-command-events.jsonl
```

只有 `central-upload` 才写 pending / acked / backoff。

### 10.3 endpoint 是占位地址

客户端会把以下地址视为未配置：

```text
https://unconfigured.dev.cx-aicode
https://unconfigured.prod.cx-aicode
https://unconfigured.staging.cx-aicode
```

占位地址不会发起网络请求，也不会记录 backoff。改成真实服务端地址后再验证。

### 10.4 服务端返回 401

含义：Bearer Token 不匹配。

检查：

- 服务端 `USAGE_API_TOKENS`
- 客户端 `CX_USAGE_UPLOAD_TOKEN`
- `.cx/config.json` 中 `tokenEnv`

### 10.5 服务端返回 413

含义：批量大小超过服务端限制。

处理：

- 降低客户端 `usageAudit.upload.batchSize`
- 或提高服务端 `USAGE_MAX_BATCH_SIZE`
- 确认 `USAGE_BODY_LIMIT` 足够

### 10.6 服务端返回 429

含义：超过限流。

处理：

- 提高 `USAGE_RATE_LIMIT_MAX`
- 按部门或项目分配多个 token
- 增大客户端 `minIntervalMs`

### 10.7 服务端返回 503

含义：数据库不可用或连接失败。

检查：

- `DATABASE_URL`
- 数据库账号权限
- 迁移是否已执行
- `/ready` 返回
- 服务端日志

### 10.8 没有 AI 行数统计

检查顺序：

1. `.cx/config.json` 中 `aiStats.enabled` 是否为 `true`。
2. `node <插件根>/hooks/usage/ai-stats-cli.js status` 是否能执行。
3. `git config --get core.hooksPath` 是否为 `.cx/hooks`。
4. `.cx/hooks/pre-commit`、`commit-msg`、`post-commit`、`pre-push` 是否存在。
5. Claude Code 会话是否通过 `/cx-*` 命令触发 AI 修改。
6. 修改文件是否被 `aiStats.include` 包含、未被 `exclude` 排除。
7. 是否执行了 `git add`，没有 staged diff 时 `pre-commit` 不会生成 stats。

### 10.9 commit message 没有 `AI-Stats-Id`

可能原因：

- `commit-msg` hook 没执行。
- `pre-commit` 没有生成 staged stats。
- staged 文件全部被 exclude。
- 提交使用了跳过 hooks 的方式。

检查：

```bash
git config --get core.hooksPath
ls .cx/hooks
node <插件根>/hooks/usage/ai-stats-cli.js status
```

### 10.10 AI 统计置信度是 LOW

常见原因：

- AI 开始前工作区已有 diff。
- 一次提交混入多个 AI 会话修改。
- 一次 AI patch 只提交了部分文件。
- merge / cherry-pick 场景。
- 会话中断后恢复。

处理建议：

- 开始 AI 工作前保持工作区干净。
- 每个 SPEC-ID 独立分支或独立提交。
- 不把人工大改和 AI patch 混在同一提交。
- 格式化提交使用 `[format]` 或 `chore(format)` 标记。

### 10.11 本地 cache 状态异常

重建索引：

```bash
node <插件根>/hooks/usage/ai-stats-cli.js rebuild-index
```

只清理过期文件：

```bash
node <插件根>/hooks/usage/ai-stats-cli.js clean
```

不要直接删除 `.cx/usage/*.jsonl`，它们是可追溯原始流水。

## 12. 安全和隐私边界

默认配置下：

- 命令统计不保存 prompt 正文。
- 命令统计不保存源码内容。
- `projectId` 是项目路径派生 ID，不是完整本地路径。
- Git 用户名和邮箱会被采集，用于团队统计归属。
- AI patch 本地会保存 diff / patch 文件，集中上报 AI patch 事件时包含变更文件列表和统计元数据；服务端 `raw_stats` / `raw_event` 会保存完整统计 JSON。

上线前建议：

- token 只通过环境变量或密钥系统注入。
- 服务端只在内网或受控网关暴露。
- 数据库按团队权限分配查询账号。
- 对 `.cx/usage/patches/` 制定本地保留和清理策略。
- 若业务代码含敏感信息，评估 AI patch 事件和 raw JSON 的字段范围后再开启集中上报。
