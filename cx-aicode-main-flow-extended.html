CX-Aicode 标准 AI 编程主流程 · 扩展版流程图

基于 cx-aicode v2.3.6 源码结构梳理：26 个命令、26 个 process skills、8 个只读 Agent、Hooks 门禁、三文件机制与质量门禁。

本版按当前使用约束处理：不把 Git/worktree 作为推荐流程；不把 cx-update 作为插件维护主路径；cx-entry 仅作为路由说明而非必需操作。

阅读方式：先看“总览流程图”，再看“cx-work 内部循环”和“Review/Verify 门禁回路”。表格列出了每个入口分支、失败分支、辅助分支和状态处理规则。

主干

Brainstorm/Spec → Approval → Work → CodeReview → Verify → Finish → Archive

硬门禁

未初始化、未审批、无 tasks、无 session marker、Review/Verify 未通过都会阻断。

Agent-first

计划、规范合规、代码质量、CodeReview、Verify、QA、DocCheck 均优先只读 Agent。

默认 TDD

cx-work 每个实现任务默认 RED → GREEN → REFACTOR，除非明确 --no-tdd。

一、标准 AI 编程主流程总览图

覆盖新需求、已有规范、Delta Spec、已有 tasks、Bug、复杂研究、辅助产物、审查失败、验证失败、完成归档等主要分支。

蓝：需求/验证/分析绿：实现/完成橙：审查/反馈红：失败/调试紫：路由/规划/规范

二、cx-work 内部每任务循环

这是实际写代码的核心流程：实现子代理必须串行；每个任务默认 TDD；规范审查必须先于代码质量审查；三文件更新不能跳过。

三、CodeReview / Verify / Feedback 门禁回路

CodeReview 判断代码质量和风险，Verify 判断是否满足设计与需求。两者互补，不能互相替代。

四、入口分支与命令决策矩阵

场景分支推荐入口主流程行为分支/例外处理
新功能 / 需求不清晰/cx-aicode:cx-brainstorm澄清目标、约束和成功标准；探索 2-3 个方案；生成 design.md + tasks.md；设计审批前禁止实现。设计未批准 → 回到方案修订；复杂任务 → cx-work-plan-reviewer 审核。
已有正式设计 / 需求说明/cx-aicode:cx-spec new --spec-id <ID> --source <file>将需求形式化为 proposal.md、design.md、tasks.md，并写入 metadata/source。validate 不通过或用户不批准 → 修订 spec。
现有系统增量变更/cx-aicode:cx-spec new --delta生成 delta-spec.md；cx-work 在 tasks.md 缺失时从 delta-spec 提取任务。REMOVED 仅归档记录，不直接生成实现任务；复杂 Scenario 拆为多任务。
已有 approved design.md + tasks.md/cx-aicode:cx-work --spec=<SPEC-ID>进入实现主流程；解析 SPEC-ID；校验 Approval；初始化三文件与 cx-work-session.json。缺审批、缺任务、多 spec 无法定位 → 阻断并提示。
Bug / 异常 / 失败/cx-aicode:cx-debug先根因调查，再模式分析、假设测试、单一修复；不得症状修复。若影响需求/架构 → 回到 brainstorm/spec；若属已批准范围 → 纳入 cx-work 修复。
复杂研究 / 长调试 / 多轮决策/cx-aicode:cx-planningFIRST、2-Action Rule、3-Strike Protocol；用于 cx-work 之外的手动规划。手动三文件不解锁编码门禁；需要实现时仍需设计审批和 cx-work。
代码结构分析/cx-aicode:cx-analyzeparser/analyzer + cx-code-analyzer 只读 Agent 解读；生成结构摘要或报告。非 Python 技术栈多为轻量扫描；生成结果需人工复核。
文档 / 图表/cx-aicode:cx-doc / cx-diagram生成模板驱动文档或 PlantUML 图表；可结合 from-code。产物是草案/模板化结果；不得把图表默认输出为 Mermaid。
自动化测试工件/cx-aicode:cx-test生成测试计划、测试用例、数据、追踪矩阵；默认可执行测试并更新 summary/feedback。RUN_TESTS=0 只生成；PROMOTE_TESTS=1 才并入项目标准测试目录。
手工 QA / 文档合规/cx-aicode:cx-qa / cx-doc-checkQA 计划、用例、回归清单、缺陷模板；文档模板合规检查。QA/DocCheck 不更新 reviewStatus/verifyStatus，不替代主质量门禁。
审查失败 / 有条件通过/cx-aicode:cx-feedback处理 CodeReview 反馈；问题多时追加 CodeReview Fix Phase。修复后必须重新跑同一 scope 的 cx-codereview。
验证失败/cx-aicode:cx-verify不能声称完成；需要新鲜验证证据，逐项验证需求/接口/数据/业务/异常。failed/conditional → 回 cx-work 修复，再重新 Review/Verify。
完成收尾 / 归档/cx-aicode:cx-finish → cx-archive检查 Review + Verify 双门禁；最终测试；本地保留/丢弃；归档规范和报告。reviewStatus/verifyStatus 缺失或失败 → finish 强制停止。
能力沉淀/cx-aicode:cx-skill-create / cx-learn从代码生成 business/tech/quality Skill；记录经验到 lessons.md。生成 Skill 必须人工审核后纳入企业规范。

五、状态分支处理规则

节点状态标准处理
实现子代理DONE进入规范合规审查。
实现子代理DONE_WITH_CONCERNS先判断 concerns 是否影响正确性/范围；影响则修复，不影响则记录并审查。
实现子代理NEEDS_CONTEXT补充上下文后重新派发，不能催促继续。
实现子代理BLOCKED判断是缺上下文、模型能力不足、任务过大还是计划错误；改变策略后再继续。
Spec Review不通过由实现子代理修复规范差距，再重新规范审查；不能直接进入质量审查。
Quality Review不通过修复质量问题，再重新质量审查；不能跳过二次审查。
CodeReview Gatepassed进入 cx-verify。
CodeReview Gateconditional_approved记录审批理由后进入 cx-verify。
CodeReview Gateconditional使用 cx-feedback 关闭、修复或升级审批后重新 CodeReview。
CodeReview Gatefailed必须修复 L1/L2 或严重问题后重新 CodeReview。
Verify Gatepassed允许进入 cx-finish。
Verify Gatefailed / conditional不得声称完成；按证据回到修复循环。

六、Hooks 与三文件机制在流程中的位置

三文件

  • .cx/work/task_plan.md：Phase/Task checkbox，cx-work 的真实任务进度源。
  • .cx/work/findings.md：研究发现、技术决策、错误与根因。
  • .cx/work/progress.md：会话日志、文件变更、测试与质量门禁记录。
  • .cx/work/cx-work-session.json：有效 cx-work 会话 marker；手动三文件不能解锁编码门禁。

Hooks

  • SessionStart：仅在有效 cx-work 会话下恢复上下文。
  • PreToolUse：静默执行初始化、审批、三文件和 session marker 准入门禁。
  • PostToolUse：在有效会话下辅助更新 progress/findings。
  • UserPromptSubmit：有效会话下注入当前任务上下文。
  • Error/Failure Hooks：记录失败，支持 3-Strike / 中断恢复。
关键边界
  • Hooks 是补充，不替代 cx-work 主流程。checkbox、progress 补充和下一任务重新读取，仍由 cx-work 主动执行。
  • cx-planning 可手动管理三文件，但不能绕过设计审批，也不能替代 cx-work-session.json。
  • QA/DocCheck/Analyze 产物默认不改变 Review/Verify 门禁状态。

八、可复制的标准主流程命令骨架

# 1. 新需求 / 需求不清晰
/cx-aicode:cx-brainstorm --spec-id <SPEC-ID> --source <需求说明.md>
# 设计生成后：可选使用 cx-work-plan-reviewer 审核；用户批准后自动/手动进入 cx-work

# 2. 已有正式需求或规范
/cx-aicode:cx-spec new --spec-id <SPEC-ID> --source <需求说明.md>
/cx-aicode:cx-spec validate <SPEC-ID>
# 用户批准 design.md + tasks.md 后：
/cx-aicode:cx-work --spec=<SPEC-ID>

# 3. 实现完成后质量门禁
/cx-aicode:cx-codereview --scope=spec --spec=<SPEC-ID> --tech=auto
# 若 conditional/failed：
/cx-aicode:cx-feedback --from-codereview --to-task-plan
# 修复后重新：
/cx-aicode:cx-codereview --scope=spec --spec=<SPEC-ID> --tech=auto

# 4. 验证与收尾
/cx-aicode:cx-verify
/cx-aicode:cx-finish
/cx-aicode:cx-archive --spec=<SPEC-ID>

# 5. 常见辅助
/cx-aicode:cx-debug --error="<错误信息>"
/cx-aicode:cx-planning --resume
/cx-aicode:cx-test --type=unit --from-code
/cx-aicode:cx-qa --plan --feature=<功能名> --write
/cx-aicode:cx-doc-check --doc=<文档.md> --template=requirements
/cx-aicode:cx-skill-create --from-code --target=business --name=<name> --path=<path>
/cx-aicode:cx-learn --type=pattern "问题 / 解决方案"
